# DuneIIScenarios

The shared headless model for the behavioural scenario harness (see `Documentation/Architecture/ScenarioHarness.md`). Depends on `DuneIISimulation` + `DuneIIWorld` + `DuneIIFormats` + `DuneIIContracts`; no rendering, no app, no oracle.

Provides: a deterministic terrain generator (real `Map_CreateLandscape`), the predefined scenario definitions (moving / close- & far-attack / guarding / move-around-building / deviate / **attackStructure** / **turretDefense** / **factoryProduce** / **repairBuilding** / **upgradeBuilding** / **sandwormEating**), a builder that lays out a `GameState` (terrain + units + optional structure + initial actions), and a runner that ticks the scenario while capturing per-tick unit snapshots. `sandwormEating` forces unit1 to a sandworm that slithers to a pickable prey (unit2) and eats it ÔÇö a visual check for the `EXPLOSION_SANDWORM_SWALLOW` gulp animation (needs `tickExplosions`, on in `scenariolab`).

The three economy scenarios drive the `tickStructure` build/repair/upgrade state machine live: `factoryProduce` (a Light Factory builds a Trike Ôćĺ credits drain + countdown Ôćĺ READY), `repairBuilding` (a damaged windtrap self-repairs, HP climbing), `upgradeBuilding` (a barracks upgrades, level 0Ôćĺ1). They set the player's no-silo credit allowance (so the House loop's storage clamp doesn't wipe the demo credits) and **settle** the structure (suspend its BUILD.EMC placement animation via a long `script.delay`) so the one-time `SetState` startup sequence doesn't overwrite the production state ÔÇö in real play the structure has long since settled before it produces. `duneii-headless` prints their per-run economy progression; `scenariolab` lists them (renders the building).

`ScenarioWorld.outcome()` (`ScenarioOutcome.swift`) reports each kind's natural endpoint as `.running`/`.finished(label)` ÔÇö a unit arrives, a building is destroyed/built/repaired/upgraded, a combatant dies, a unit is deviated. It's a **visual-harness affordance** (the `scenariolab` "finished" banner + auto-pause), not a simulation victory/game-over model; the sim models no game-over. `ScenarioWorld` carries `kind` + `structureSlots` for this.

The builder takes both the **unit** script (`UNIT.EMC`) and the **structure** script (`BUILD.EMC`), and the runner ticks via the real `Simulation.tick()` (the full four-phase loop ÔÇö unit + structure scripts + the House economy). Combat is fully ported, so units move/aim/**fire/impact** and structures run their scripts: `attackStructure` shows a tank destroying a (pre-weakened) building Ôćĺ its death branch (Explode Ôćĺ Destroy Ôćĺ Structure_Remove + soldiers) runs live; `turretDefense` shows a gun-turret's script acquire Ôćĺ aim Ôćĺ fire at an approaching enemy. The `scenariolab` macOS app renders these for visual assessment (re-blitting structure changes); `ScenariosTests` verify layout + that `moving` advances + that the structure scenarios damage/destroy + the turret fires, and `ScenarioGoldenTests` asserts whole trajectories bit-for-bit vs the OpenDUNE scenario dump (incl. `attack-structure`).
