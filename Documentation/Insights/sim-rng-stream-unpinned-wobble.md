# A green movement golden does NOT prove random256-stream parity ÔÇö `wobble` draws are invisible

**Finding:** Our `Random256` stream can be byte-misaligned with the oracle's and a full-trajectory movement golden will still pass, because the per-step `Unit_Move` `wobbleIndex` draw (`Tools_Random_256() & 7`) only affects the rendered sprite ÔÇö never `position`/`orient`/`hp`/`targets` ÔÇö so the snapshot never compares it. The misalignment stays hidden until a native that *does* steer a compared field off the same stream lands. `Script_Unit_IdleAction` (native `0x31`) is the first such case: its level-select `Tools_Random_256() & 1` picks turret-vs-body, so an off-by-one stream flips the guard's idle twitch (oracle rotates the turret ÔåÆ body `orient` stays 0; ours rotated the body ÔåÆ divergence).

**Why it matters:** Don't read "the movement golden matches end-to-end" as "our RNG is in lockstep with OpenDUNE." It only proves the *deterministic* (RNG-independent) trajectory matches. The oracle advances `random256` during scenario setup (`Map_CreateLandscape`, `Scen_LoadUnit` unit init) and via every mover's `wobble`, so by tick 1 its state is already past the seed-0 origin (oracle's first post-setup draw was `0x13`, while our `Random256(seed:0).next()` is `0x80`). Chasing byte-exact alignment means replicating *all* of those draws ÔÇö which is byte-exact RNG-order parity, explicitly **not** the project bar (CLAUDE.md "What not to do" + Plan ┬º0 Principle 1: stochastic behaviour matches only within seed-to-seed spread).

**Evidence:** `Script_Unit_IdleAction` port `Code/Frameworks/DuneIISimulation/Script/Natives/UnitScriptFunctions.swift` (`idleAction`); the wobble draw `Code/Frameworks/DuneIISimulation/UnitMovement.swift:102-105`; OpenDUNE `src/script/unit.c:1760` (`Script_Unit_IdleAction`) + `src/unit.c` (`Unit_Move` wobble). Diagnosed by adding LCG/256 tracing to `Parity_RunScenario` (`Repositories/OpenDUNE/src/parity.c`) and running `--parity-scenario=96 ÔÇª --parity-lcg-trace= --parity-random-trace=`: 20 ticks of the guard scenario drew only 4 random256 bytes (`t1 0x13 NULL`, `t6 0xC2 u22`, `t6 0xF6 u22`, `t9 0x55 u23`) and 2 LCG values (both 0). The guard golden now gates at its deterministic prefix (`Code/Tests/ScenariosTests/ScenarioGoldenTests.swift`, `guard` `compared: 6`).

**How to apply:** When a stochastic native lands on the GUARD/idle/combat path, gate the affected scenario golden at the tick *before* its first RNG draw rather than trying to byte-match the spread; verify the native itself by a focused decision-trace test (same draw order/count vs OpenDUNE) instead. Reach for the oracle's `--parity-lcg-trace=`/`--parity-random-trace=` (now wired into `Parity_RunScenario`) to see exactly when/what the oracle draws before deciding a divergence is a bug vs. unpinnable spread.

**Corroborated (`attack-rocket`, two stationary Launchers):** with *no* unit moving, nothing draws the wobble, so the stream stays aligned and *even both GUARD units' `IdleAction` twitches match the oracle byte-for-byte* ÔÇö the exact same `IdleAction` that desyncs `guard`. Conclusion: idle rotation is only "stochastic spread" relative to us because a *different* moving unit's wobble shifts the shared stream; remove the wobble source and idle RNG is perfectly reproducible. So the fix for byte-matching idle/combat scenes is to avoid (or pin) wobble draws, not to treat idle rotation as inherently unmatchable.
