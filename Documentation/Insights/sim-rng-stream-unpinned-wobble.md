# A fully-seeded golden divergence is a real draw-count bug — trace-align, don't write it off as "spread"

**Finding (corrected 2026-05-30):** With a fixed seed and a faithful transcription, our `Random256`/`RandomLCG` streams **must** match the oracle's draw-for-draw — same logic ⇒ same number and order of draws. So a per-tick golden divergence under identical seeds + the same headless harness is **always a concrete discrepancy**, never inherent non-determinism. The recurring `guard` (tick 6) / `attack-rocket` (tick 69) divergences turned out to be a **single real bug**: our `GameLoop_Team` early-returned (`guard let runner = teamScript else { return }`) when no `TEAM.EMC` was bridged, **skipping the cursor re-arm and its `Tools_Random_256() & 7` draw** — but OpenDUNE re-arms `s_tickTeamGameLoop` (drawing that byte) *unconditionally* every fire, teams or not (`team.c:22`). So our shared `Random256` stream was one byte behind from tick 1 onward.

**Why it stayed hidden so long:** the dropped draw only changes a *compared* field when a later native *reads* the shifted stream. Stationary combat (`attack-close`) reads no `Random256` (tank bullets aren't `notAccurate`), so it matched the full 400 ticks despite the drift. `moving`/`move-trike` likewise. The drift first surfaced where the stream is read: `guard`'s sitting tank runs `Script_Unit_IdleAction` (0x31) at tick 6 and reads `0xC2/0xF6` — but at the *wrong stream offset*, flipping its idle twitch; `attack-rocket`'s `notAccurate` scatter (`Tile_MoveByRandom`) reads it at tick 69 and lands 1 orientation-unit off. **The earlier "render-only wobble spread / don't chase RNG order" diagnosis was wrong** — the tell was the `ctx=NULL` draw at tick 1 (`t1 idx0 0x13 NULL`): a *phase-level* draw (no `g_scriptCurrentUnit`), i.e. the team-loop cursor, not unit setup or wobble.

**Fix:** make the team-loop cursor re-arm (+ its draw) unconditional, gating only the team-script *execution* on the runner — `Simulation.gameLoopTeam` (`Code/Frameworks/DuneIISimulation/Simulation.swift`). `guard` and `attack-rocket` both became **full 400-tick matches** (`ScenarioGoldenTests` `compared: 0`).

**How to find this class of bug:** when a fully-seeded golden diverges, dump both engines' draws — the oracle's `--parity-random-trace=` / `--parity-lcg-trace=` (wired into `Parity_RunScenario`, tagging each draw `tick/idx/byte/ctx`) — and find the **first divergent draw by index**. A `ctx=NULL` (or `t<team>`) draw the other engine doesn't make points at a *phase-level* draw (team-loop cursor, a per-tick maintenance draw) rather than a per-unit one. The number of draws per tick is the invariant to check first: if the counts differ, that's the bug — fix the missing/extra draw, don't gate the golden around it.

**Caveat that remains real:** a green *deterministic-trajectory* golden (positions/hp) still does not by itself *prove* RNG lockstep — it only proves the RNG-independent state matches. But the cure is to trace-align and fix the real draw discrepancy (as here), **not** to assume the divergence is unpinnable and gate early. Truly unpinnable draws would require a genuinely non-deterministic input, which a seeded headless run does not have.
