# A structure's position is the tile corner, not the centred sub-tile

**Finding:** a structure stores `o.position` as the tile **corner** — the low byte (the `0x80` sub-tile offset) zeroed — whereas a unit stores the **centred** sub-tile. `Structure_Place` does `s->o.position = Tile_UnpackTile(position); s->o.position.x &= 0xFF00; s->o.position.y &= 0xFF00;` (`structure.c`). Units keep the `Tile_UnpackTile` centre (`| 0x80`).

**Why it matters:** the 128px (half-tile) offset is invisible to anything that re-packs the position (`Tile_PackTile` strips the sub-tile, so map stamping / occupancy / `packed`-keyed lookups are unaffected) — but it shifts every **distance and direction** calc that targets a structure. A unit aiming its turret at a 2×2 structure computes `Tile_GetDirection(unit.position, indexGetTile(structure))`; with the structure centred instead of cornered, the aim direction lands a few orientation units off, so the unit finishes rotating later and **fires several ticks late** (or at the wrong sub-tile). Centred-vs-cornered is internally self-consistent, so unit tests that don't compare against the oracle won't catch it.

**Evidence:** OpenDUNE `Structure_Place` (`src/structure.c`, the `&= 0xFF00`); our fix in `DuneIIWorld/Scenario/ScenarioLoader.swift` `loadStructure`. Caught by the `attack-structure` scenario golden (`Tests/ScenariosTests/ScenarioGoldenTests.swift`): the oracle dumped the windtrap at `position (4608,4096)` = corner, ours at `(4736,4224)` = centred, and our tank fired ~10 ticks late until the position was cornered — after which the scenario matches the oracle for the full 400 ticks.

**How to apply:** whenever you set a structure's `o.position` from a packed tile, strip the sub-tile (`& 0xFF00` on both axes), never the plain centred `Tile32.unpack`. This is the structure analog of "units centre." Any new structure placement path (save converter, a future `Structure_Create`, test helpers used for oracle comparison) must do the same.
