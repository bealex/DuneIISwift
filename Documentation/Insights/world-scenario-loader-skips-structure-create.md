# The scenario loader hand-rolls structure init and skips Structure_Create's setup

**Finding:** `ScenarioLoader.loadStructure` allocates a structure and sets only house/position/state/hitpoints — it does **not** call `Structure_Create`, so it skips that function's `upgradeTimeLeft` arming (player factory → 100, AI factory → max `upgradeLevel` then 0), the Harkonnen-light-vehicle `upgradeLevel = 1`, and the `linkedID`/`objectType` sentinels. OpenDUNE's `Scenario_Load_Structure` (`scenario.c:204`) *does* call `Structure_Create(...)` and only then overrides hitpoints/state/degrades.

**Why it matters:** a loaded construction yard (or any placed factory) had `upgradeTimeLeft = 0`, so the GUI's `canUpgrade` gate (`upgradeTimeLeft != 0`) was always false — the Upgrade option never appeared, even on a campaign where the CY is upgradable. The cross-engine goldens don't catch it: no golden scenario upgrades within its 400 ticks, and `upgradeTimeLeft` draws no RNG, so the divergence is observably inert there (which is *why* it survived). It only surfaces in live GUI play.

**Why fixing it in `loadScenario` is risky:** `structureIsUpgradable` reads `campaignID` and `playerHouseID`, which the client sets *after* `loadScenario` (and the parity harness may not set at all). Arming inside the loader would compute against the wrong globals and could perturb the parity path. So the fix is a separate `GameState.armPlacedFactoryUpgrades()` called from `GameModel.load` once those globals are set — the parity `loadScenario` path stays byte-identical.

**Evidence:** `DuneIIWorld/Scenario/ScenarioLoader.swift:189` (loadStructure) vs `DuneIISimulation/UnitCombat.swift:352` (`structureCreate`); fix `DuneIIWorld/State/GameState+Lifecycle.swift` (`armPlacedFactoryUpgrades`) called from `DuneIIClient/GameModel.swift` load; test `SimulationTests/StructureBuildTests.armsPlacedFactoryUpgrades`; OpenDUNE `scenario.c:204` (`Scenario_Load_Structure`) + `structure.c:373` (`Structure_Create`).

**How to apply:** when a "missing" structure property shows up only in the client (upgrade arm, a creator-house take-over field, a build-link sentinel), suspect the hand-rolled `loadStructure` skipped a `Structure_Create` step — don't assume the field is computed lazily. Arm such state in a post-load client pass (after campaignID/playerHouseID are set), not inside the golden-verified `loadScenario`. Related: [[host-presentation-gap-not-sim]].
