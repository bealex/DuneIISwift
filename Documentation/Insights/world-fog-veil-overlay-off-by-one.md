# Our stored fog-veil overlay tile id is one higher than the oracle's (gameplay-neutral)

**Finding:** at scenario load a veiled cell's `overlayTileID` (the fog veil, `g_veiledTileID` ≈ `iconMap[iconMap[ICM_FOG=7] + 16]`) is stored as **124 in our `GameState.map`** but **123 in OpenDUNE**. Surfaced by the first golden to ever compare map-tile state (`wall-destruction`/`slab-indestructible`), which dump the cell every tick — every prior scenario golden compared only units/structures/houses.

**Why it matters (and why it's low-priority):** it is **gameplay-neutral**. `Tile_IsUnveiled(t) = t > veiled || t < veiled-15` gives the *same* result for the actual stored values whether veiled is 123 or 124 (a revealed 0 is unveiled either way; the stored veil value is itself veiled either way), so fog reveal, targeting, and every unit/structure golden are unaffected — which is why it stayed hidden. But it means you **cannot** compare the raw `overlayTileID` veil value cross-engine; the `wall`/`slab` tile goldens compare `groundTileID` only (the destroyed-wall ground revert is the real observable). Root cause is unconfirmed — likely a one-off in the fog-of-war icon-group base lookup or the `MapGenerator` veil store (`map[i].overlayTileID = UInt8(truncatingIfNeeded: tileIDs.veiled)`); both 123 and 124 fit in the 7-bit field, so it is a value difference, not a truncation.

**Evidence:** `wall-destruction`/`slab-indestructible` goldens (`ScenarioGoldenTests`, `TileGolden` deliberately omits `overlay`); our store at `MapGenerator.swift:183`; the lookup `TileIDs.swift` (`tileID(group: 7, offset: 16)`). Oracle: `g_veiledTileID = g_iconMap[g_iconMap[7] + 16]` (`parity.c`), `Tile_IsUnveiled` (`sprites.c:477`).

**How to apply:** if you ever need to compare the fog overlay value cross-engine (e.g. a renderer fog golden against the oracle), pin this down first. Until then, assert tile **ground** ids, not overlay, in map-state goldens. When adding a `GameState` field that mirrors an OpenDUNE bitfield, check the field width (7-bit `overlayTileID`) — don't assume a `UInt8` store round-trips identically.
