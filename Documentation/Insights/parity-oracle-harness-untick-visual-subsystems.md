# The oracle scenario harness ticks only the four GameLoop phases — not Explosion/Animation

**Finding:** OpenDUNE's scenario parity harness (`src/parity.c` `Parity_RunScenario`) runs **only** `GameLoop_Team/Unit/Structure/House` per tick — it never calls `Explosion_Tick` (and would not call any other visual subsystem the full `GameLoop_Main` ticks). The full game ticks them; the parity scenario runner does not.

**Why it matters:** any subsystem we tick on our side that **draws RNG** (the shared `Random_256` / `RandomLCG`) but the oracle harness doesn't will desync every golden that triggers it. `Explosion_Tick` draws `Tools_RandomLCG_Range` (`SET_RANDOM_TIMEOUT`) and `Tools_Random_256` (`TILE_DAMAGE`, `SET_ANIMATION`); `attack-structure` destroys a windtrap → an `EXPLOSION_STRUCTURE` → an LCG draw. **`Animation_Tick` is the same hazard** — its `.pause` command draws `Tools_Random_256() % 4` (`GameState+Animation.swift`), and a placed windtrap/silo's animation hits it within seconds. We *shipped* `animationTick` unticked-gated-incorrectly (running it unconditionally) on the wrong assumption that "no golden reaches an RNG-drawing animation command in its compared window" — the **RNG-stream golden** (below) proved that wrong: `economy` and `attack-structure` each made 6–12 extra `Random256` draws vs the oracle, all from `animationTick`. The *start* calls (`Explosion_Start`/`Animation_Start`) are safe — RNG-free, and the oracle does call them.

**Evidence:** `Repositories/OpenDUNE/src/parity.c:436-439,730-733` (phase list, no `Explosion_Tick`/`Animation_Tick`); `explosion.c:233,92,183` + `GameState+Animation.swift` `.pause` (the RNG draws); our gates `Code/Frameworks/DuneIISimulation/Simulation.swift` (`tickExplosions` + `tickAnimations`, both default `false`). Caught by the **RNG draw-stream golden**: `ScenarioGoldenTests` asserts our per-tick `Random256`/`RandomLCG` stream is byte-identical to the oracle's `--parity-random-trace`/`--parity-lcg-trace` (`RngTraceSink`) — see `sim-rng-stream-unpinned-wobble`.

**How to apply:** when porting a visual/secondary subsystem that draws RNG (explosions, animations, map blooms, sandworm shimmer, …), gate its per-tick advance behind a flag **off** on the golden/oracle-matched path and **on** only in the visual apps — or teach `parity.c` to tick it too and regenerate. Wiring the RNG-free *start* into the deterministic path is fine; *ticking* is the hazard. And **assert the draw stream**, don't trust the state golden — a visual-subsystem RNG draw is invisible to a state golden until it shifts a compared field, but the draw-stream golden catches it the instant it happens.
