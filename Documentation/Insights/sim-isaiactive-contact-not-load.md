# `isAIActive` is a contact flag in the real game, pinned only by the parity harness

**Finding:** OpenDUNE's `House.flags.isAIActive` ("the house has been seen by the human, everything now becomes active — team attack, house missiles, rebuilding") is **not** set at scenario load in the real game. It is set lazily by `Unit_HouseUnitCount_Add` (unit.c:2657) the first time a non-winger enemy unit is sighted — waking **both** the seer's house and the seen unit's house. The OpenDUNE **`parity.c` oracle** (and our `ScenarioLoader`) pin it on at load *only because the parity harness doesn't tick fog* (its comment: "in a real game that bit is set when the AI first sees an enemy; the parity harness pins it on at load").

**Why it matters:** copying the harness pin into the live game makes the AI fully active from tick 0 — it ramps its economy/rebuild and (crucially) its **palace house-missile auto-fire** (`!human && isAIActive`) immediately, so it assaults the player base far too early. The contact-gated path was already wired in our sim (`unitUpdateMap` sets `isAIActive` when any unit stands on an `isUnveiled` tile; `mapUnveilTile` does it on a fresh reveal — both call `unitHouseUnitCountAdd`, GameState+Lifecycle.swift:886-887), so the pin was the *only* thing forcing premature aggression. (Note: the live `duneii` doesn't even bridge `TEAM.EMC`, so team attacks don't run there — the pin's real effect was production + missiles, not team raids.)

**Evidence:** fix — `ScenarioLoader.loadScenario(activateTeamHousesAI:)` (default `true` for the parity harness/goldens; `GameModel.load` passes `false`); contact path `GameState+Lifecycle.swift:886-887` + `:957`, `GameState+Fog.swift:26`; OpenDUNE `unit.c:2657` (`Unit_HouseUnitCount_Add`), `parity.c` `Scen_LoadTeam`. Tests: `TeamCreateTests` (default pins; game-mode leaves the house dormant). Scenario goldens use the default ⇒ byte-identical.

**How to apply:** when porting a behaviour the parity harness sets up at load (a pinned flag, a forced state), check whether the **real game** establishes it differently (lazily, on an event) — `parity.c` often shortcuts things the harness can't drive (fog, input, timing). The shared `loadScenario` should keep the harness default so goldens stay oracle-matched; the live client opts into the real-game path via a parameter.
