# The script VM runs on a COPY of the engine — natives that mutate the live script were clobbered

**Finding:** the unit/structure script runners execute the EMC VM on a **value-type copy** of the engine (`var engine = state.units[slot].o.script` / `…structures[slot]…`), then write it back at the end. The dispatched natives, however, mutate the entity's **live** script through `state` — `Object_Script_Variable4_Clear/Set/Link` (the unit↔structure var-4 reservation link) and `Unit_SetAction` (reloads the script: new PC + `variables`). Because the VM's copy was stale, those mutations were **invisible to later opcodes in the same batch and clobbered on write-back**. OpenDUNE shares one `&script` pointer, so the VM and natives always agree; our copy diverged.

The visible symptom was the **harvester economy stalling**: a harvester would dock at the refinery and refine its spice into credits, but the refinery's deploy path (`Script_Structure_Unknown0A81` clearing its dangling var-4) was undone on write-back, so the deploy native (`Unknown0C5A`) never fired and the harvester was **stuck inside the refinery forever**. The 400-tick scenario goldens never ran a full ~9000-tick harvest→return→refine→deploy cycle, so this was never caught.

**Fix:** reconcile the copy with live state around each native dispatch (`StructureScriptRunner.run` / `UnitScriptRunner.run`): sync VM→state **before** the call (native sees current values), and **after**, if a native rewrote the live script (`state…o.script != before`), adopt it back into the VM — otherwise keep the VM engine (eng-param natives like `moveToTarget`/`calculateRoute` set `delay`/`scriptPC` on the local `eng`, never on `state`). This makes the harvester **dock, refine, and redeploy** (was permanently stuck) with **no golden regression** (476 tests green).

**Evidence:** `Frameworks/DuneIISimulation/Script/Natives/StructureScriptRunner.swift` + `UnitScriptRunner.swift` (the `let before = eng; state…script = eng; …dispatch…; if …script != before { eng = …script }` blocks). Test: `HarvesterCycleTests.fullCycle` — a real-`UNIT.EMC`/`BUILD.EMC` scenario that drives the whole loop; it asserts harvest→dock→refine→redeploy and currently `withKnownIssue`s the very last leg.

**Known remaining gap (re-diagnosed 2026-05-31):** the deployed *empty* harvester resumes STOP (`Unit_SetActionDefault → actionsPlayer[3]`) instead of HARVEST. **The reconciliation is *not* the cause** — traced through `UnitScriptRunner.run`: `setActionDefault` loads the STOP script into the VM's `eng` and `state.actionID = STOP`; nothing writes `state.o.script` during dispatch, so `eng` (STOP) is the final write-back — *consistent*. (The previously-suspected `state.units[slot].o.script = engine` write-back inside `setAction(engine:in:)` is a behavioral **no-op** for this leg — don't bother.) So the harvester *genuinely reaches* `SetActionDefault`; the bug is a **native-return divergence in the post-deploy RETURN-action EMC branch** — a native returns a value that takes the STOP branch where OpenDUNE takes the `Unit_SetAction(ACTION_HARVEST)` branch. Note `ACTION_HARVEST` is **never** set by C (`grep ACTION_HARVEST src/*.c` only *checks* it) — the resume-HARVEST is purely an EMC `SetAction` in the harvester's RETURN script, gated by a native. Pin it with a **`--parity-script-unit` cross-engine per-opcode trace** of the harvester across the full ~9000-tick loop (needs an INI harvester scenario + the harvest order added to the oracle harness), find the first divergent native return, fix that native. Do **not** patch the branch by hand (project rule: trace-align first).

**How to apply:** any time a native mutates `state…o.script` (variables, PC via `SetAction`, var-4 links), remember the VM holds a *copy* — the change must be reconciled, or it vanishes. When a native legitimately writes **both** the local `eng` and the live script, the simple compare-and-adopt is insufficient; unify it onto one storage. Don't trust the 400-tick goldens to cover multi-thousand-tick economy loops — write a dedicated scenario test (like this one).
