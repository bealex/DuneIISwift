# The script VM runs on a COPY of the engine ‚Äî natives that mutate the live script were clobbered

**Finding:** the unit/structure script runners execute the EMC VM on a **value-type copy** of the engine (`var engine = state.units[slot].o.script` / `‚Ä¶structures[slot]‚Ä¶`), then write it back at the end. The dispatched natives, however, mutate the entity's **live** script through `state` ‚Äî `Object_Script_Variable4_Clear/Set/Link` (the unit‚Üîstructure var-4 reservation link) and `Unit_SetAction` (reloads the script: new PC + `variables`). Because the VM's copy was stale, those mutations were **invisible to later opcodes in the same batch and clobbered on write-back**. OpenDUNE shares one `&script` pointer, so the VM and natives always agree; our copy diverged.

The visible symptom was the **harvester economy stalling**: a harvester would dock at the refinery and refine its spice into credits, but the refinery's deploy path (`Script_Structure_Unknown0A81` clearing its dangling var-4) was undone on write-back, so the deploy native (`Unknown0C5A`) never fired and the harvester was **stuck inside the refinery forever**. The 400-tick scenario goldens never ran a full ~9000-tick harvest‚Üíreturn‚Üírefine‚Üídeploy cycle, so this was never caught.

**Fix:** reconcile the copy with live state around each native dispatch (`StructureScriptRunner.run` / `UnitScriptRunner.run`): sync VM‚Üístate **before** the call (native sees current values), and **after**, if a native rewrote the live script (`state‚Ä¶o.script != before`), adopt it back into the VM ‚Äî otherwise keep the VM engine (eng-param natives like `moveToTarget`/`calculateRoute` set `delay`/`scriptPC` on the local `eng`, never on `state`). This makes the harvester **dock, refine, and redeploy** (was permanently stuck) with **no golden regression** (476 tests green).

**Evidence:** `Frameworks/DuneIISimulation/Script/Natives/StructureScriptRunner.swift` + `UnitScriptRunner.swift` (the `let before = eng; state‚Ä¶script = eng; ‚Ä¶dispatch‚Ä¶; if ‚Ä¶script != before { eng = ‚Ä¶script }` blocks). Test: `HarvesterCycleTests.fullCycle` ‚Äî a real-`UNIT.EMC`/`BUILD.EMC` scenario that drives the whole loop; it asserts the whole loop harvest‚Üídock‚Üírefine‚Üíredeploy‚Üíharvest-again directly (the last leg was un-`withKnownIssue`d 2026-06-02 ‚Äî see below).

**Former "remaining gap" ‚Äî RESOLVED 2026-06-02: it was a *test-scenario artifact*, not an engine bug.** The deployed-empty harvester reaching `STOP` was faithful. A per-opcode trace of our harvester (instrument `UnitScriptRunner.run` to dump `pc`/native/return for the harvester in the post-deploy window) showed it redeploys already in `ACTION_HARVEST`, runs the HARVEST routine, `Unit_Harvest` fails (not on spice), `SearchSpice(3)` finds nothing, **the escalating wider search (radius 20/40/60) is skipped, then `SetAction(STOP)`**. Reading the UNIT.EMC: that escalation is gated `(!targetMove) && (GetInfo(0x0E) != variables[3])` = (no move target) && (unit house ‚âÝ player house) ‚Äî i.e. **the wide spice search is for *AI* harvesters only; a *player* harvester only searches radius 3** and otherwise STOPs. Every piece was verified against OpenDUNE source ‚Äî `Script_Unit_GetInfo` 0x0E = `Unit_GetHouseID` (`script/unit.c:946`), `targetLast` cleared on dock in `Unit_Move` (`unit.c:1493`), and the refinery deploy sequence (`RefineSpice`‚Üí`FindUnitByType`‚Üí`Unknown0C5A`, BUILD.EMC type 12). So a player harvester redeployed with **no spice within 3 tiles** genuinely stalls in OpenDUNE too. The test had the refinery **6 tiles from the spice**, so the deploy tile was out of range ‚Üí STOP. **Fix: build the refinery *adjacent* to the spice** (as a player actually does), so the spice-nearest deploy tile lands on/next to spice and the radius-3 search succeeds. The loop now closes; `HarvesterCycleTests.fullCycle` asserts `harvestedAgain` directly (no `withKnownIssue`). **Lesson:** the prior "native-return divergence" hypothesis was wrong ‚Äî before hypothesising an engine bug from a failing *scenario* test, instrument the actual opcode path and check the scenario geometry is one the faithful engine supports.

**How to apply:** any time a native mutates `state‚Ä¶o.script` (variables, PC via `SetAction`, var-4 links), remember the VM holds a *copy* ‚Äî the change must be reconciled, or it vanishes. When a native legitimately writes **both** the local `eng` and the live script, the simple compare-and-adopt is insufficient; unify it onto one storage. Don't trust the 400-tick goldens to cover multi-thousand-tick economy loops ‚Äî write a dedicated scenario test (like this one).
