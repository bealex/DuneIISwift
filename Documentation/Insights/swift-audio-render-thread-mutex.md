# An AVAudioSourceNode render block must be a nonisolated @Sendable closure — capture no actor-isolated self

**Finding:** the render closure passed to `AVAudioSourceNode(format:renderBlock:)` runs on the audio IO thread. If that closure inherits any actor isolation — e.g. it's a closure literal written inside a `@MainActor` method/init and it captures `self` (even `[weak self]`) — it **compiles fine** but **crashes at runtime** the moment audio starts, with `_dispatch_assert_queue_fail` → `_swift_task_checkIsolatedSwift` → `swift_task_isCurrentExecutor` deep in `AVAudioSourceNode pullInputBlockFromRenderBlock`. AVFAudio invokes the isolated closure through a bridging thunk that asserts "am I on the right executor?", and the audio thread is not the main queue, so it traps. The cure is to make the closure genuinely *nonisolated* and `@Sendable`: it must capture **only `Sendable` values and never `self`**.

**Why it matters:** the crash is in `libdispatch`/`libswift_Concurrency` with no line of your code in the top frames, so it reads like an OS/AVFAudio bug rather than an isolation mistake. And it's invisible to the build *and* to headless tests (a CI box has no output device, so the render thread never starts) — it only fires when a real device drives the node. The two-`Mutex`/sending dance you do to satisfy the *compiler* (see below) does nothing to prevent this; isolation of the closure itself is a separate axis.

**How to get a nonisolated render block (the working shape):**
1. Put all cross-thread state in a `Sendable` **box** — a `final class … : Sendable` holding a `Mutex<State>` (`import Synchronization`; a class may store the noncopyable `Mutex`). Add a `Mutex<(@Sendable (T) -> Void)?>` "notifier" so the render thread can hand work back to the owner *without capturing it*.
2. Build the node in a `nonisolated static func makeNode(box:…)` so the closure literal lives in a nonisolated context and captures only the box. The render body is a `nonisolated static func` taking the box.
3. The `@MainActor` owner sets `box.notifier` to `{ [weak self] x in Task { @MainActor in self?.handle(x) } }` *after* init — that closure is `@Sendable` and only hops to the actor; the render block calls it (once, guarded), never touching the actor directly.

**Also (the compiler axis, distinct from the crash):** on a `@MainActor` class, `static let`/`static func` are main-actor-isolated too — mark every render-path member `nonisolated`. Construct non-`Sendable` instances (the OPL3 chip/driver) *inside* the `withLock` closure so no local lingers and the `Mutex` value stays sendable across the boundary.

**Evidence:** `Code/Frameworks/DuneIIAudio/ADLMusicPlayer.swift` — `RenderBox: Sendable`, `makeNode`/`render`/`install` all `nonisolated static`/`nonisolated`, render block captures only `box`. The first cut captured `[weak self]` in the block and crashed on launch with the trace above. Tests: `AudioTests/ADLMusicTests` (build/wiring + real-data render; the isolation crash itself is only reproducible with a live audio device, so it's a manual `swift run duneii` check).

**How to apply:** for any real-time C-style callback (AVAudioSourceNode/AVAudioSinkNode/CoreMIDI/`AudioUnit` render block) owned by an actor-isolated type: capture a `Sendable` box, never `self`; create the closure in a nonisolated context; route every actor callback through a `@Sendable` notifier that does `Task { @MainActor … }`. Treat "compiles" as nothing — exercise it on a real device.

**Second, separate lesson — don't synthesize on the audio thread, and watch the build mode.** Heavy DSP (an emulated OPL3 FM chip) is *far* too expensive for the render callback — it misses the deadline and the audio glitches/drops out. Split it producer/consumer: a background thread runs the synth and fills a ring buffer ahead of playback; the callback only copies out of the ring (or emits silence on underrun). And measure throughput per build mode: the OPL3 emulation renders at **~1.9× real-time in a debug Swift build** (slower than playback — *no* architecture can fix that; the ring just perpetually underruns) but **~0.17× in release**. DSP-grade audio code needs `-O`; document that the host must be run optimized (`swift run -c release …`) for it to work. Evidence: `ADLMusicPlayer` (producer thread + `Ring`); throughput measured directly off `OPL3Chip.generateResampled()`.
