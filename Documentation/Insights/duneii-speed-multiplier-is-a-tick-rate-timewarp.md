# The duneii client speed multiplier is a pure tick-rate time-warp, distinct from OpenDUNE's gameSpeed

**Finding:** there are **two unrelated "speeds"** in play, and conflating them leads to wrong conclusions about how fast things happen:
- **`GameModel.gameSpeed`** (the toolbar 0.5×/1×/2×/4×) is the *host* multiplier. It only scales how many `sim.tick()`s run per real second — `GameScene.update`: `tickAccumulator += dt * 60 * model.gameSpeed`, then `advance(ticks: steps)` runs that many ticks. It never touches sim state (nothing assigns `state.gameSpeed` from the host). It's a deterministic-sim *time-warp*.
- **`GameState.gameSpeed`** (OpenDUNE's 0–4 config) is a *gameplay* tuning consumed by `Tools_AdjustToGameSpeed` — it sets per-tick cadences (the structure/house tick intervals, the build-speed cap, AI build slowdown). It stays at its default; the toolbar does not change it.

**Why it matters:** any system driven by `timerGame` (it increments once per `sim.tick()`) automatically scales with the toolbar multiplier *for free*, because the multiplier changes how fast `timerGame` advances in wall-clock. Building construction is the canonical example: `structureTickStructure` advances a build's `countDown` by `buildSpeed` on each structure tick (fired off the `timerGame` cursor `+ adjustToGameSpeed(30,15,60,…)`), with **no wall-clock anywhere** — so a build takes a fixed number of *sim ticks*, and 2× simply elapses them in half the real time. The same is true of movement, combat, economy, animations: they're all tick-driven, so they stay in lockstep at every multiplier. You don't (and must not) special-case the multiplier into individual subsystems.

**How to apply:** to make something honor the toolbar speed, make it tick-driven (advance on `timerGame`/a sim cursor) — that's all. Don't reach for `state.gameSpeed` to "go faster"; that's the orthogonal OpenDUNE cadence knob. When verifying "does X respect the speed multiplier?", confirm X is purely tick-driven and that `GameScene.update` scales `steps` by `model.gameSpeed` (it does). Caveat: `steps = min(steps, 16)` per frame is an anti-spiral cap that only triggers under a severe frame-rate drop and slows the *whole* sim uniformly (not any one subsystem). Evidence: `GameScene.update` (`:121`/`:124`), `Simulation.tick` (`timerGame &+= 1`), `gameLoopStructure`/`structureTickStructure`. Related: `parity-oracle-harness-untick-visual-subsystems`.
