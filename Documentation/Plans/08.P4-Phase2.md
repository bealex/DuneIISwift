# 08. P4 Phase 2 — data tables + naive movement + sim-to-visual sync

Status: Landed 2026-04-20. Supersedes the UnitInfo / sim-to-visual / SetDestination items in `07.P4-Phase1.md` §4 (1, 6, 7). P4 remains open; see §4.

## 1. What shipped

### 1.1 `Simulation.UnitInfo` — full 27-entry port

Trimmed to the fields our wired host functions + naive movement need. Source: `src/table/unitinfo.c`, hand-extracted field-by-field from the Dune II 1.07 table.

Fields per unit: `hitpoints`, `fireDistance`, `fireDelay`, `damage`, `movementType`, `hasTurret`, `explodeOnDeath`, `movingSpeedFactor`, `turningSpeed`, `actionsPlayer[4]`, `actionAI`. Plus a new `Simulation.MovementType` enum (`foot`/`tracked`/`harvester`/`wheeled`/`winger`/`slither`) and `Simulation.ActionID` namespace (the 14 action constants).

`Simulation.UnitInfo.lookup(_:)` returns nil for out-of-range type IDs.

### 1.2 `Simulation.StructureInfo` — 19-entry port

Fields: `hitpoints`, `buildCredits`, `fogUncoverRadius`, `layout`. Plus `Simulation.StructureLayout` enum (1x1 through 3x3) with a `dimensions` computed property returning `(width, height)`. Source: `src/table/structureinfo.c`.

### 1.3 Expanded `Script_Unit_GetInfo`

The 20-way switch now covers 0x00 (hitpoints ratio, `hp * 256 / maxHP`), 0x02 (`fireDistance << 8`), 0x0B (has-destination), 0x0D (`explodeOnDeath`), 0x10 (turret orientation — falls back to body since we don't separate yet), and 0x12 (`movementType & 0x40` — always 0 in vanilla data). Only 0x13 (seenByHouses-bit) still requires `g_playerHouseID` plumbing.

### 1.4 `SetSpeed` scenario-scale

`makeSetSpeedUnit` now applies OpenDUNE's `byScenario` check: units NOT placed by the scenario get their speed down-scaled by 192/256. The `UnitSlot.byScenario` flag is plumbed through both WorldSnapshot init paths (save → `ObjectHeader.flags.byScenario`; scenario → `true` always).

### 1.5 `Script_Unit_SetDestination` + `SetDestinationDirect`

- `makeSetDestinationUnit(host:)` — writes `peek(1)` to `targetMove` if valid, else clears. Harvester/refinery check deferred.
- `makeSetDestinationDirectUnit(host:)` — writes `targetMove` and faces the target using `Pos32.direction`. Used by carryalls.

Wired into the unit function table at slots 0x05 and 0x19.

### 1.6 Naive movement step

`Simulation.Scheduler.tick()` now has a new `tickNaiveMovement()` pass that runs **before** script dispatch. For every allocated unit with `targetMove != 0`:

1. Resolve target pos32.
2. If `|dx| + |dy| ≤ 32 px`, snap to target and clear `targetMove` (arrival).
3. Otherwise step by `max(4, speed/4)` pixels, proportional to the longest axis; face the direction of travel.

**This is not OpenDUNE-faithful.** Real movement runs on tile-granular `route[]` arrays filled by the pathfinder, applied per-tick by `Unit_MoveStep`, with collision against the tile grid. The naive step ignores obstacles, doesn't use the `speedRemainder` accumulator, and slides diagonally where OpenDUNE would step axis-aligned through the route buffer. It's a placeholder so users see *something moving* before pathfinding lands.

Replaces cleanly in P4 Phase 3: swap `tickNaiveMovement` for a proper `Unit_MoveStep` once the pathfinder fills `route[]`.

### 1.7 Sim-to-visual sync

`ScenarioScene` now:

- Builds unit / structure markers **from the live `scheduler.host` pools**, not from `scenario.units/structures` directly. Keyed by pool index.
- Adds `syncVisualsFromPool()` — called after every tick — that refreshes each marker's `position` from the slot's `positionX`/`positionY` (pos32, converted to SKScene coordinates via `screenPositionPos32`), and hides markers whose slots have been freed (`Die`, `Destroy`).
- Structure markers use the real footprint from `StructureInfo.layout.dimensions` — 3×2 refineries are now 3×2 tiles wide, not 2×2 like they were in Phase 1.

Net effect: any position / orientation change the scripts make, or any arrival from the naive movement step, is **visible** on the scenario scene each tick.

### 1.8 Tests

**301 tests across 41 suites, all green** — 14 new tests:

- `UnitInfoTests` (7): table size, canonical unit lookups (Carryall / Tank / Harvester / Sandworm), out-of-range, structure table size + lookups, GetInfo 0x00 / 0x02 / 0x0D.
- `NaiveMovementTests` (5): SetDestination valid / invalid, tick advances position, tick arrives + clears, stays-put when no targetMove.
- `EmcHostBatch67Tests` +2: SetSpeed byScenario no-scale vs. non-scenario down-scale.

## 2. What you should see

`swift run duneii` → scenario scene:

- The `Tick N` HUD still advances.
- Unit / structure markers are positioned exactly where the scenario places them (same as Phase 1).
- **If any script calls `SetDestination` on a unit**, that unit slides toward its target each tick. Most mission-1 units start in `ACTION_GUARD` so they won't move until something triggers a destination write — but sandworms (`ACTION_HUNT`) and AI-controlled units running `Unit_IdleAction` may call into `SetDestination` once `IdleAction` + pathfinding lands.
- If scripts call `Die` or `Destroy`, the marker disappears (hidden).
- Structure markers now match the real footprint (3x2 refineries etc.).

**You may still see no motion** — that's because the subset of host functions that would lead to a `SetDestination` call (`FindBestTarget`, `MoveToTarget`, `IdleAction` + random-position picker, AI team decisions) isn't wired yet. The loop *is* alive; scripts are just stuck in their idle branches.

Quick way to force-verify movement works: modify a script manually or add a debug call that writes `host.units[0].targetMove = Scripting.EncodedIndex.unit(1).raw` on tick 1. The unit will slide toward unit 1.

## 4. P5 — first previews

With Phase 2 done, P5 has two items that are now approachable without deeper P4 work:

- **Real structure footprints on the map grid** — we now have `StructureInfo.layout`. Stamping 3×2 refineries / 2×2 factories as real tile sprites (not coloured rectangles) is a pure-rendering task on top of existing data.
- **Unit SHPs with per-house palette remap** — requires a SHP-frame-per-unit-type lookup + the 16-entry house palette table (`src/table/houseinfo.c`'s `minimapColor` + `paletteShift`). Moderate effort; lands after pathfinding so we can see unit SHPs with proper orientations.

Both are deferred to P5; this plan focuses on P4.
