# 07. P4 Phase 1 — host-function wiring + live script execution

Status: Landed 2026-04-20. Supersedes the host-function-wiring item in `06.P3-Complete-P4-Kickoff.md` §3 (1). P4 remains open; see §4 for the work ahead.

## 1. What shipped

### 1.1 Pool slot expansion

`Simulation.UnitSlot` gained eight new fields; `Simulation.StructureSlot` gained three. All are plumbed through both `WorldSnapshot` init paths (save-based and scenario-based).

| Slot | Field | Source (save path) | Source (scenario path) |
|---|---|---|---|
| Unit | `positionX`, `positionY` | `ObjectHeader.positionX/Y` | `Pos32.centered(at: spawn.position)` |
| Unit | `hitpoints` | `ObjectHeader.hitpoints` | `spawn.hitPoints` |
| Unit | `seenByHouses` | `ObjectHeader.seenByHouses` | `0` |
| Unit | `speed` | save-record `speed` | `0` |
| Unit | `spriteOffset` | save-record `spriteOffset` | `0` |
| Unit | `blinkCounter` | save-record `blinkCounter` | `0` |
| Unit | `inTransport` | `ObjectHeader.flags.inTransport` | `false` |
| Structure | `positionX`, `positionY` | `ObjectHeader.positionX/Y` | `Pos32.centered(at: spawn.position)` |
| Structure | `hitpoints` | `ObjectHeader.hitpoints` | `spawn.hitPoints` |

### 1.2 `Pos32` + tile math

New `Code/Core/Sources/DuneIICore/Map/Tile.swift` exports a pure-Swift `Pos32` value type (OpenDUNE's `tile32`, pixel-scale coordinates where each map tile = 256 pixels) plus four static operations:

- `Pos32.centered(at: PackedPosition)` — tile centre.
- `Pos32.distance(_:_:)` — `longest + shortest / 2` (port of `Tile_GetDistance`).
- `Pos32.direction(from:to:)` — 0..255 heading (full port of `Tile_GetDirection` with the 32-entry `directions[]` table and the 4-quadrant offset table from `src/tile.c:342`).
- `Pos32.of(_:host:)` — resolves an `EncodedIndex` to a `Pos32` against live pool state.

Tested with four tests covering tile-centre math, cardinal directions, the distance metric's three branches, and pool resolution across all four `EncodedIndex` kinds.

### 1.3 Host enrichment

`Scripting.Host` gained a `voiceLog: [VoicePlay]` sink paralleling `textLog`. `VoicePlay` is a new nested struct holding `(voiceID, positionX, positionY)` — one entry per `Script_General_VoicePlay` call.

### 1.4 Batch 6 — position-dependent host functions

Three new factories on `Scripting.Functions`:

- `makeGetDistanceToTile(host:)` — distance from current object to the encoded tile, or `0xFFFF` invalid.
- `makeGetDistanceToObject(host:)` — same metric under a different name (OpenDUNE separates them for different C call sites; at this layer they're identical).
- `makeVoicePlay(host:)` — appends a `VoicePlay` at the current object's position.

### 1.5 Batch 7 — simple mutators (no pathfinding, no combat)

Eight unit and one structure mutator:

- `makeSetOrientationUnit(host:)` — writes `orientationCurrent`, returns new value.
- `makeSetSpeedUnit(host:)` — clamps 0..255, writes `speed`.
- `makeStopUnit(host:)` — writes `speed = 0`.
- `makeBlinkUnit(host:)` — writes `blinkCounter = 32`.
- `makeDieUnit(host:)` — frees the slot (economy / score updates deferred).
- `makeSetSpriteUnit(host:)` — writes `spriteOffset = -(peek(1) & 0xFF)`.
- `makeSetTargetUnit(host:)` — writes `targetAttack`, faces target via `Pos32.direction`; clears on invalid.
- `makeIsInTransportUnit(host:)` — `1`/`0` read of the `inTransport` flag.
- `makeDestroyStructure(host:)` — frees the structure slot (explosion / debris deferred).

### 1.6 `Scripting.Functions.unitTable` + `structureTable`

Two factory functions that return fully-populated 64-slot `[VM.Function?]` arrays mirroring OpenDUNE's `g_scriptFunctionsUnit` / `g_scriptFunctionsStructure` tables at `src/script/script.c:33,64`. Every wired slot is commented with its OpenDUNE function name; unported slots are left `nil` with an explicit comment explaining the blocker (pathfinding / projectile pool / explosion pool / UnitInfo table / etc.).

**Coverage of `g_scriptFunctionsUnit`**: 28 of 64 slots wired. Unit-category-specific: `GetInfo`, `SetAction`, `SetOrientation`, `Blink`, `Die`, `SetSprite`, `Stop`, `SetSpeed`, `IsInTransport`, `GetAmount`, `SetTarget`. Generics shared with other tables: `DisplayText`, `GetDistanceToTile`, `GetOrientation`, `IsEnemy`, `Delay`, `IsFriendly`, `NoOp` (×4), `RandomRange`, `FindIdle`, `VoicePlay`, `GetLinkedUnitType`, `GetIndexType`, `DecodeIndex`, `UnitCount`, `DelayRandom`, `GetDistanceToObject`.

**Coverage of `g_scriptFunctionsStructure`**: 11 of 25 slots wired. `Delay`, `NoOp` (×5), `SetState`, `DisplayText`, `GetState`, `VoicePlay`, `Destroy`.

### 1.7 `ScenarioScene` integration

`ScenarioScene.setUpScheduler` now wires the real function tables into the VMs. `RandomSource` is seeded from the scenario's map seed for reproducibility. The unit VM loads `UNIT.EMC` and the structure VM loads `BUILD.EMC` from the install; scripts now execute real opcodes instead of halting on the first `SCRIPT_FUNCTION` call.

### 1.8 Tests

**287 tests across 39 suites, all green.** 18 new tests:

- `TileMathTests` (4) — centre math, distance metric, cardinal directions, pool resolution.
- `EmcHostBatch67Tests` (14) — each new host function; invalid-reference paths; function-table slot assignments.

## 2. What each tick loop does now

Starting from the main menu, clicking through to the scenario scene now produces a live simulation loop on the user's screen:

1. `Simulation.WorldSnapshot(scenario:resolver:)` populates the pools from SCENA001.INI spawns (positions in pixel coords, hitpoints from the scenario, types via `.typeID` mapping).
2. `Scripting.Host` wraps the pools + empty text / voice logs.
3. `Scripting.Functions.unitTable(host:source:)` / `structureTable(host:source:)` produce fully-wired function tables.
4. Unit / structure VMs load `UNIT.EMC` / `BUILD.EMC` from the install.
5. `ScenarioScene.update(_:)` ticks the scheduler every 5 frames.
6. On each tick, every allocated slot's engine runs up to 7 (unit) / 3 (structure) opcodes. Scripts can now genuinely execute — reading targets, setting orientations, checking distances, queueing voice callbacks into `host.voiceLog`.

What's **not** visible yet: unit movement (requires pathfinding + `SetDestination` / `CalculateRoute`), combat (requires projectile pool + `Fire`), production (requires `RefineSpice` + credit flow), fog-of-war visibility updates.

## 3. Why some slots aren't wired

Each unported slot has a dependency that's its own project:

| Dependency | Blocks | Estimated scope |
|---|---|---|
| Pathfinding (`Script_Unit_Pathfinder` / `Unit_GetTileEnterScore`) | `SetDestination`, `MoveToTarget`, `CalculateRoute`, `SetDestinationDirect`, `Pickup`, `TransportDeliver` | ~500 lines + tests |
| Projectile pool | `Unit_Fire`, `Unit_ExplosionSingle`, `Unit_ExplosionMultiple`, `Structure_Fire` | ~200 lines + tests |
| Explosion pool | Same + `Structure_Explode`, saboteur death FX | ~150 lines + tests |
| `g_table_unitInfo` | `SetActionDefault`, `GetInfo 0x00` (hitpoints ratio), `GetInfo 0x02` (fireDistance), `Unit_Rotate`, `SetSpeed`-scaling | ~200 lines (data table port) |
| `g_table_structureInfo` | `Structure_RefineSpice`, `Structure_FindUnitByType` | ~150 lines |
| `g_playerHouseID` on Host | `GetInfo 0x13` (seenByHouses-bit), `Unit_DisplayDestroyedText`, `Unit_MoveToStructure` player checks | 1 Host field + threading |
| Fog-of-war grid on Host | `Unit_RemoveFog`, `Structure_RemoveFogAroundTile` | 1 Host field + tests |
| Map search (`Map_SearchSpice`) | `General_SearchSpice`, `Unit_Harvest` | ~100 lines |
| Unit spawn primitive (`Unit_Create`) | `Unit_RandomSoldier`, `Unit_CallUnitByType`, `Unit_MCVDeploy` | ~100 lines |
| Team AI (`Script_Team_*`) | Entire team category | ~300 lines + team pool |

The three big ones for "first winnable scenario" are **pathfinding**, **projectile pool + Fire**, and **UnitInfo table**. With those landed, most of the unit / structure AI becomes real.

## 4. P4 — remaining work

Ordered by impact:

1. **UnitInfo table + structureInfo table ports** — unblocks `GetInfo` subcases, `SetActionDefault`, `Fire` damage / fireDistance, `Rotate`. Mechanical port from `src/table/unitinfo.c` / `src/table/structureinfo.c`. ~1 session.
2. **Pathfinding** — port `Script_Unit_Pathfinder` + `Unit_GetTileEnterScore` + map-based cost queries. Unlocks unit movement end-to-end. ~2-3 sessions with tests.
3. **Projectile pool + `Unit_Fire` / `Structure_Fire`** — new `ProjectilePool` + per-frame flight tick + collision against the UnitPool. ~1-2 sessions.
4. **Explosion + damage propagation** — tile-radius damage, explosion pool, `Map_MakeExplosion`. ~1 session.
5. **Team AI** — port `Script_Team_*` (`src/script/team.c`) + `TeamPool` + `teamaction.c` weighted-action table. ~1-2 sessions.
6. **Tick parity harness** — capture a golden OpenDUNE run for mission 1 (via a small OpenDUNE patch that dumps pool state to JSON each tick), diff our tick output against it. Closes the §6 "simulation parity is non-negotiable" goal. ~1 session.
7. **Sim-to-visual sync** — once units actually move, update the marker `SKShapeNode.position` from the pool each tick in `ScenarioScene`. Trivial once the pools are genuinely changing.

Realistic estimate: **P4 is 6-10 more focused sessions**. This plan's Phase 1 closed the "tick loop is alive" gap; phases 2 and 3 (UnitInfo + pathfinding) are the heaviest.
