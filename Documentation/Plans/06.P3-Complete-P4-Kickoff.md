# 06. P3 Complete · P4 kickoff — tick loop on the scenario scene

Status: Landed 2026-04-20. Supersedes the P3 section of `01.Initial.md` §8 and opens P4.

## 1. What shipped

### 1.1 P3 — scenario walker with unit / structure markers

`ScenarioScene` now renders three layers on top of the generated 64×64 ground grid:

| Layer | Source | Shape | Colour |
|---|---|---|---|
| Ground tiles | `ScenarioWorld.map[x, y].groundTileID` → `SKTexture` via cached ICN images | 16×16 sprite | per-tile palette |
| Structure markers | `scenario.structures` where `!isGenerated` | 32×32 rounded box, 55 % alpha, 1 px white stroke | `HouseColors.color(for:)` |
| Unit markers | `scenario.units` | circle r = 6.4 px | `HouseColors.color(for:)` |

Slabs and walls (`isGenerated`) stay as part of the tile grid (stamped by `ScenarioWorld`), not as marker sprites — they have no pool presence.

`HouseColors` is the on-screen palette per `House` case (canonical per-house hues): Atreides blue, Harkonnen red, Ordos green, Fremen yellow, Sardaukar purple, Mercenary orange. Placeholder — real units apply per-house palette remap baked into their SHP frames; that's a P5+ pass.

### 1.2 P4 kickoff — live world + tick loop

Three pieces bridge a loaded scenario into a running simulation:

1. **`Scenario.{House,UnitType,StructureType}.typeID`** — extension properties returning OpenDUNE's `HOUSE_*` / `UNIT_*` / `STRUCTURE_*` numeric IDs (`src/house.h`, `src/unit.h`, `src/structure.h`). Ports the type-enum boundary between our string-keyed scenario model and the byte-keyed pool layer.

2. **`Simulation.WorldSnapshot.init(scenario:resolver:)`** — fresh-snapshot path (no save involved):
   - Map baseline from `Map.Generator.generate(seed: scenario.mapField.seed, resolver:)`.
   - Houses: one `HousePool` slot per key in `scenario.houses` (at `house.typeID`).
   - Units: `UnitPool` slot per `scenario.units[i]`, allocated at slot `i`, with `type` / `houseID` / `orientationCurrent` set from the spawn.
   - Structures: `StructurePool` slot per `scenario.structures[i]` where `!isGenerated` — `GEN*` rows become map stamps, not pool slots. Reserved slots `79..81` are untouched.
   - Tiles: baseline only (no sparse overrides — there's no save chunk here).
   - Test: synthetic Atreides-MCV + Harkonnen-Tank + Atreides-ConstYard + GEN-slab scenario asserts `snap.units.findArray == [0, 1]`, `snap.structures.findArray == [0]`, and that `GEN*` slabs are filtered from the pool. One test, zero install dependency.

3. **`ScenarioScene.update(_:)`** drives a tick every 5 frames (≈12 Hz at 60 fps, matching OpenDUNE's main-loop cadence):
   - `setUpScheduler(scenario:resolver:)` builds a `Simulation.WorldSnapshot(scenario:resolver:)`, wraps its pools in a `Scripting.Host`, and constructs a `Simulation.Scheduler` with VMs that load `UNIT.EMC` + `BUILD.EMC` from the install (function tables empty — scripts halt on first `SCRIPT_FUNCTION` opcode, which is the expected failure mode for now).
   - Every `framesPerTick` (= 5) frames, `scheduler.tick()` runs the full per-slot dispatch loop; a HUD label refreshes with `Tick N · units M · structures K`.
   - `Formats.Emc.Program.empty` is a new canonical zero-instruction program used when the install doesn't ship a given EMC.

### 1.3 Scaffolding touch-ups

Minor public-API additions to make scenarios composable by tests:

- `Scenario.init()` — zero-valued default, used by tests that build scenarios from scratch.
- `Scenario.HouseLayout.init(…)`, `Scenario.UnitSpawn.init(…)`, `Scenario.StructureSpawn.init(…)` — explicit public memberwise inits. The auto-synthesised ones were internal.
- `Formats.Emc.Program.empty` — canonical no-op program for VM construction fallback paths.

## 2. Known visual gaps (expected, not bugs)

The full P3 spec from `01.Initial.md` §8 reads "place units/structures statically." We've done that with markers. Real units / structures need SHP frames + per-house palette remap + (for structures) multi-tile footprints. That's P5 visual polish.

- Unit markers are circles, not unit sprites.
- Structure markers are 2×2-tile coloured rectangles regardless of the structure's real footprint (some are 2×3 in the original, e.g. refinery).
- No fog of war rendering. Every tile is visible.
- No spice overlay sprite (spice shows via the underlying ground tile's colour when `Map.Generator` stamps a spice tile).

## 3. P4 — what's next

P4's "Simulation" phase is now a live canvas. Scripts run but halt on first host call. Path forward:

1. **Plumb remaining host functions** into the scheduler's VMs. Batches 2-5 (`DisplayText`, `UnitCount`, `GetOrientation`, `Is{Enemy,Friendly}`, `DelayRandom`, `GetIndexType`, `DecodeIndex`, `GetLinkedUnitType`, `FindIdle`, `GetInfoUnit`, `SetActionUnit`, `GetAmountUnit`, `GetStateStructure`, `SetStateStructure`) already exist — they just need wiring into the `unitVM` / `structureVM` function slots per the `g_scriptFunctions{Unit,Structure}` tables at `src/script/script.c:33,64`.
2. **Position-dependent generics** — batch 6: `GetDistanceToTile` / `GetDistanceToObject` + `positionX`/`positionY` on pool slots.
3. **Unit action mutators** — `Script_Unit_SetDestination`, `Script_Unit_Fire`, `Script_Unit_CalculateRoute`, etc. Needs pathfinding + projectile pools.
4. **Visual sync** — once units move under script control, redraw their markers from the pool each tick (not from the scenario spawn list).
