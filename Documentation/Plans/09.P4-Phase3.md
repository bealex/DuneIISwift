# 09. P4 Phase 3 — pathfinder + route-follower

Status: Landed 2026-04-20. Supersedes the pathfinding / `Unit_MoveStep` / `CalculateRoute` / `MoveToTarget` items in `08.P4-Phase2.md` §3. P4 remains open; see §3.

## 1. What shipped

### 1.1 `Simulation.LandscapeInfo` — 15-entry port

All 15 landscape types (`LST_NORMAL_SAND`..`LST_BLOOM_FIELD`) with their 6-entry `movementSpeed[MovementType]` array plus `letUnitWobble` / `isValidForStructure` / `isSand` / `canBecomeSpice` flags. Source: `src/table/landscapeinfo.c`.

`Simulation.LandscapeInfo.lookup(_ landscape: LandscapeType)` returns the row indexed by the enum's raw value.

### 1.2 `Simulation.Pathfinder` — faithful OpenDUNE port

Full port of `Script_Unit_Pathfinder` + `Script_Unit_Pathfinder_Connect` + `Script_Unit_Pathfinder_Smoothen` from `src/script/unit.c:1012..1286`. Plus the tile-packed helpers `packedTile(x:y:)`, `packedDistance(from:to:)`, `directionPacked(from:to:)` (the 16-entry `returnValues` table from `src/tile.c:193`).

**Public API**:

```swift
enum Simulation.Pathfinder {
    static let mapDirection: [Int32]          // 8 tile-index deltas
    typealias TileEnterScore = (_ packed: UInt16, _ orient8: UInt8) -> Int32
    struct Route { var buffer: [UInt8]; var score: Int32; var size: Int }
    static func findRoute(src: UInt16, dst: UInt16, bufferSize: Int, score: TileEnterScore) -> Route
}
```

**Algorithm shape** (matches OpenDUNE — see inline `src/script/unit.c` citations):

- Walks greedy toward `packedDst` via `directionPacked / 32`.
- When the direct step hits a tile with score > 255, spawns two `Pathfinder_Connect` attempts (clockwise + counterclockwise) looking for a connection to the next walkable tile on the direct line.
- Picks the lower-cost partial route (or whichever succeeded) and copies steps into the main buffer.
- Re-smooths the complete route via `Pathfinder_Smoothen` — collapses opposite-direction pairs, merges 45°-adjacent turns, folds 90°-turn-plus-step-through pairs.

Route buffer is a sequence of 3-bit direction indices (0..7) terminated by `0xFF`. Matches OpenDUNE's `u->route[14]` byte-for-byte.

### 1.3 `UnitSlot.route` + `currentDestination` fields

Two new fields on `Simulation.UnitSlot`:

- `route: [UInt8]` (14 entries, initialised `0xFF` — mirrors `u->route[14]`).
- `currentDestinationX / currentDestinationY: UInt16` — pos32 of the unit's current-step target. Zero when idle. Mirrors `u->currentDestination`.

Save-path plumbing: `WorldSnapshot(loading:baseline:)` copies `slot.route` from the save record when the array is the expected 14 entries.

### 1.4 `Scripting.Host.tileEnterScore` closure

`Host` grew an optional score function:

```swift
var tileEnterScore: ((_ packed: UInt16, _ orient8: UInt8, _ movementType: Simulation.MovementType) -> Int32)?
```

When `nil`, `CalculateRoute` falls back to "every tile walkable at cost 128" — useful for tests and early sessions. Live `ScenarioScene` wires a real map-backed scorer in `makeTileEnterScorer`: looks up the tile's landscape via `TileResolver.landscapeType(...)`, reads `LandscapeInfo.movementSpeed[movementType]`, applies the diagonal tax (`speed -= speed/4 + speed/8` when `orient8 & 1 != 0`), and inverts to get a cost.

### 1.5 `Script_Unit_CalculateRoute` (slot 0x0C)

Full port of `src/script/unit.c:1296`. When `route[0] == 0xFF`, runs the pathfinder, copies up to 14 steps into `route[]`, clears `targetMove` on no-route. When already mid-route, truncates if we're close to `dst`. Returns 1 while unwalking, 0 on arrival.

Uses the unit's `UnitInfo.movementType` to pick the correct scorer branch — tracked units see sand as cheaper than rock, wingers ignore terrain entirely (all walkable).

### 1.6 `Script_Unit_MoveToTarget` (slot 0x16)

Minimal port — returns `1` when within the scheduler's arrival threshold (32 px Manhattan) of the targetMove tile, `0` otherwise. Motion itself is handled by the scheduler's route-follower.

### 1.7 Scheduler route-follower

Replaces `tickNaiveMovement` with a route-aware mover:

1. On each tick, for every unit with `currentDestination != 0` OR `route[0] != 0xFF` OR `targetMove != 0`:
   - If `currentDestination == 0` and `route[0] != 0xFF`: populate `currentDestination` with the pos32 centre of the tile reached by applying `route[0]`'s direction to the unit's *current* tile. This is stable — the subsequent "current tile" shifts as the unit crosses the boundary don't move the goalpost.
   - If `currentDestination == 0` and `targetMove != 0`: fall back to moving straight toward the encoded tile (same as Phase 2's naive step).
2. Step `max(4, speed/4)` pixels toward the goal, facing it.
3. On arrival (manhattan ≤ 16 px), snap. If we arrived via a route step, pop `route[0]` (memmove-left) and clear `currentDestination` so the next tick pulls the next step. If we arrived via `targetMove`, clear it.

**Key bugfix vs. Phase 2**: the earlier `tickNaiveMovement` recomputed the sub-goal every tick, so as a unit's `positionX` crossed a tile boundary mid-step, the sub-goal jumped forward one tile and arrival was never detected. `currentDestination` locks the goal until the unit arrives.

### 1.8 Tests

**308 tests across 42 suites, all green.** 7 new tests in `PathfinderTests`:

- `packedTile` converts pos32 pixel coordinates to tile indexes.
- `packedDistance` uses the OpenDUNE longest+shortest/2 metric.
- `directionPacked` returns 0x00 / 0x40 / 0x80 / 0xC0 at cardinals.
- `findRoute` produces a non-empty 5-step route on an open 64×64 map.
- `findRoute` signals "no path" via `buffer[0] == 0xFF` on a fully-impassable scorer.
- `findRoute` routes around a 7-tile-long vertical wall without crossing it.
- Scheduler route-follower advances a Trike along a 2-step route, terminating when both steps are consumed.

Plus the existing `NaiveMovementTests` was updated to use a closer target matching the new 16-px arrival threshold.

## 2. What you should see

`swift run duneii` → scenario scene:

- Pre-existing behaviour unchanged: HUD `Tick N` advances, markers in scenario positions, structures honour footprints, clicking returns.
- When any script calls `SetDestination` + `CalculateRoute` on a unit, that unit now **follows a real pathfinder-computed route around obstacles** instead of sliding straight.

Whether you see motion on mission 1 still depends on which script branches fire — most mission-1 units idle in `ACTION_GUARD` and never call `CalculateRoute` unless woken by combat or team AI. But the simulation is now correct for any unit that *does* run through `SetDestination` → `CalculateRoute` → (per-tick) route-follower.
