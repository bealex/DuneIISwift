# Tick Parallelization & Optimization — plan

**Status: first experiment measured (2026-06-02, branch `parallelization-experiment`); see §8.** A benchmark (`Apps/simbench`) + a measurement-grade parallel unit phase (`Simulation+Parallel.swift`, non-golden) now exist and answer the headline question — **sequential wins, by a lot**; naive parallelism is currently net-negative on this engine. The design below stands; §8 records what the numbers say to do first. This is the plan for making `Simulation.tick()` faster, in two tracks the user asked for: (1) a separate **`SPEEDUP` compilation mode** (a build parameter) that (2) removes all non-essential debug output, plus the broader parallelization/optimization analysis that mode enables. Read `Architecture/Overview.md` and `Architecture/ParityHarness.md` first; this assumes the four-phase tick (`Frameworks/DuneIISimulation/Simulation.swift`).

There are **three build tiers**, and they coexist — choosing one never removes the others:

1. **Default (no flag)** — the faithful, fully-instrumented build. Trace sinks live; goldens and RNG-stream parity hold. This is the verification build and the default for all correctness work. Always available.
2. **`SPEEDUP`** — same logic, same RNG order, observation stripped + `-O`. Behaviorally **identical** to the default (the §2.5 equivalence golden proves it); just faster. Always available alongside the default.
3. **`SPEEDUP_UNORDERED`** (new, this revision) — builds on `SPEEDUP` but **deliberately abandons the RNG-draw-order contract** so the whole tick can be parallelized. It is **explicitly not golden** — it trades bit-exact parity for throughput. It is a *separate key*; turning it on changes nothing about tiers 1–2, which remain the parity builds. See §2.6 and §4e.

## 0. The non-negotiable invariant (read before anything else)

**Determinism + behavioral parity are the acceptance bar for *every* change here** (core principles 1 and 5) — **with exactly one explicit, opt-in exception: the `SPEEDUP_UNORDERED` key (§2.6).** Same scenario + seed + command stream ⇒ byte-identical run; and the run stays bit-identical to the OpenDUNE oracle wherever behavior is deterministic. Concretely, two things must stay byte-identical after any optimization or parallelization:

- **the scenario goldens** (`ScenarioGoldenTests`, regenerated by `Scripts/gen-scenario-goldens.sh`), and
- **the RNG draw-stream golden** — our per-tick `Random256`/`RandomLCG` draws, in number and order, match the oracle's (see `RngTraceSink` + insight `sim-rng-stream-unpinned-wobble`).

This is not a guideline, it is the test. Any change that flips a golden byte **in a parity build (tiers 1–2)** is wrong **by definition** — it has reordered or dropped an RNG draw, raced a shared mutation, or altered logic. The goldens are the safety net that makes aggressive optimization safe to attempt.

**The one exception** is the `SPEEDUP_UNORDERED` build (§2.6), which is *defined* by giving up the RNG-order contract and is therefore *expected* to diverge from the goldens. It carries a different acceptance bar (§5.2): memory-safety and statistical "within OpenDUNE's seed-to-seed spread," never byte-identity. Crucially, this exception **never touches the default or plain-`SPEEDUP` builds** — they keep the golden bar above, unchanged. The trace/RNG-stream goldens are always generated in a parity build; `SPEEDUP_UNORDERED` is never the build that certifies logic.

## 1. The determinism ceiling on parallelism

The ordered tick is *mostly inherently sequential*, and it is important to be honest about why, so we don't chase parallelism that can't preserve parity:

- **One shared RNG stream, order-significant.** `Random256` and `RandomLCG` are drawn in a single global order across all four phases and all entities. The oracle draws the *same number in the same order*; that order is the parity contract. Any scheme that lets two entities draw concurrently reorders the stream and diverges. So **every code path that draws RNG must run in the existing sequential order** — this rules out naive "parallel-for over all units." *This is precisely the constraint `SPEEDUP_UNORDERED` (§2.6, §4e) chooses to drop:* once you accept divergence from the oracle, the RNG order stops being a contract and the whole tick opens up to parallelism. The other two ceilings below remain even there — they are about memory safety, not parity.
- **One shared map + shared pools.** Per-entity work writes the map (`unitUpdateMap`, spice, overlays) and reads/writes other entities (targets, houses, the object pools). Concurrent writers race. **This ceiling is *not* lifted by `SPEEDUP_UNORDERED`:** giving up RNG order makes results non-deterministic, but a true data race on a shared `Array`/pool in Swift is undefined behavior (corruption/crash), not merely non-determinism. So even the unordered tier must make shared access *memory-safe* — by sharding entities so writers don't overlap, or by locking the few shared structures — it just no longer has to keep it *ordered* (see §4e).
- **Fixed phase order** Team → Unit → Structure → House, each iterating a deterministic `PoolFind` cursor.

What *is* independent, and therefore parallelizable **while staying golden**, falls into three buckets (§4a–§4d): work that draws no RNG and touches only its own entity; subsystems that are entirely off the golden/oracle path; and *whole separate simulations*. The single biggest single-sim speedups for parity builds, though, come from **§2 (SPEEDUP) and §3 (single-thread)**, not from threading one tick. The one path to *full* single-tick threading is the non-golden `SPEEDUP_UNORDERED` tier (§4e), which buys throughput by spending parity.

## 2. The `SPEEDUP` compilation mode (requirements 1 + 2)

### 2.1 What it is

A compile-time configuration that produces a faster binary by (a) **stripping all observation/debug instrumentation** to nothing and (b) building optimized. It is selected by a build parameter, never at runtime, so the stripped code has *zero* footprint (not even a branch).

### 2.2 Mechanics — how the flag is passed

Two viable mechanisms; we recommend the second for a first-class named build:

- **Ad-hoc:** `swift build -c release -Xswiftc -DSPEEDUP`. `-D SPEEDUP` defines the `SPEEDUP` compilation condition for every Swift target; `-c release` gives `-O`. No manifest change. Good for a quick A/B.
- **Recommended — env-driven manifest.** `Package.swift` reads an env var and conditionally injects the define (and any extra flags) into the engine targets, so a single documented command builds the mode:

  ```swift
  // Package.swift (top)
  import Foundation
  let speedup = ProcessInfo.processInfo.environment["SPEEDUP"] == "1"
  let speedupSettings: [SwiftSetting] = speedup ? [.define("SPEEDUP")] : []
  // …then on DuneIIWorld / DuneIISimulation (and any target with hot-path tracing):
  //   swiftSettings: speedupSettings
  ```

  Built with `SPEEDUP=1 swift build -c release` (wrap it in `Scripts/build-speedup.sh`, alongside the repo's `TMPDIR`/`xcrun`/`--disable-sandbox` quirks). The env-var form keeps the manifest the single source of truth and lets SPEEDUP also carry optimizer flags (§2.4) without retyping `-Xswiftc`.

The source uses `#if !SPEEDUP` to keep the instrumentation in normal builds and compile it out under SPEEDUP.

### 2.3 What it strips (requirement 2 — "remove all non-essential debug output")

The simulation has **no `print`/`Logger`/`os_log`/`#if DEBUG` in its hot path** (audited). "Debug output" here is precisely the **parity/trace instrumentation**, all currently runtime-gated by a `nil` optional (cheap, but not free, and a maintenance surface). SPEEDUP removes the hooks *and* the fields they test:

- **RNG trace hooks.** `Random256.next()` (`Rng/Random256.swift:39`) and `RandomLCG.range()` (`Rng/RandomLCG.swift:44`) end with `traceSink?.recordR256/LCG(...)`. Under SPEEDUP, wrap the `public var traceSink: RngTraceSink?` property *and* the call in `#if !SPEEDUP`, so each draw is pure arithmetic with no optional load/branch. (These draws happen thousands of times per tick — this is the one hot-loop hook worth removing.)
- **Script decision-trace.** The per-opcode `if let tracer … ScriptTraceLine.decode … tracer.record` in `StructureScriptRunner.run` (`Script/Natives/StructureScriptRunner.swift:73`) — and the equivalent unit/team paths if/when added — wrap in `#if !SPEEDUP`. `ScriptTraceLine.decode` re-decodes every opcode for the trace; gone under SPEEDUP.
- **Any future inline trace `Log`.** Standing memory rule is "every sim mutation gets a trace `Log`." When such a logging seam is added, route it through a single `@inline(__always)` shim that is `#if !SPEEDUP` (a no-op body under SPEEDUP), so the rule and the speed mode don't fight.

The EMC disassembler, `assetgen`, and the test/oracle harnesses are **not** in any shipped tick — they are unaffected and need no guarding.

### 2.4 Optimizer settings SPEEDUP may carry

- **`-O` (release): mandatory.** This is the single largest win. (For calibration: the OPL3 synth measured ~1.9× real-time in debug vs ~0.17× in release — a ~11× swing; the sim is the same kind of arithmetic-heavy Swift and benefits similarly.) SPEEDUP should always be built `-c release`.
- **`-Ounchecked`: deliberately out of scope.** It would remove bounds and overflow traps, but the sim's bounds/overflow traps guard *pool-index* and *array* access, where removing them turns a real porting bug into silent memory corruption instead of a clean trap. We keep the code safe: **all builds — including SPEEDUP and `SPEEDUP_UNORDERED` — retain the checks.** `-O` is the optimizer ceiling here; we do not trade safety for the marginal trap-elision gain.

### 2.5 The SPEEDUP correctness rule + its golden

**SPEEDUP must be behaviorally identical to a normal build** — it may only *remove observation*, never change logic, math, or RNG order. The acceptance test is a **flag-on/flag-off equivalence golden**: run a fixed scenario for N ticks in both a normal release build and a SPEEDUP build, and assert the final `GameState` (or its serialized `SaveGame` hash) is byte-identical. Because SPEEDUP strips the trace sinks, the *trace-based* goldens (RNG-stream, script decision-trace) necessarily run in **non-SPEEDUP** builds — SPEEDUP is verified by the final-state equivalence, the trace goldens verify the logic, and the two together cover it.

## 2.6 The `SPEEDUP_UNORDERED` key — drop RNG-order parity to parallelize the whole tick

### 2.6.1 What it is and why it exists

`SPEEDUP_UNORDERED` is a **second, stronger compile-time key** for the use cases that want maximum throughput and *do not* need bit-exact parity — bulk AI rollouts, parameter sweeps, search/self-play, "just simulate millions of ticks fast." It builds on `SPEEDUP` (implies it: `-O`, all observation stripped) and then **deliberately relaxes the one constraint that forces the tick to be sequential — the global RNG draw order (§1).** Once draw order is no longer a contract, the entity loops can run concurrently, so this key is the gate for the *full intra-tick parallelism* described in §4e.

It is, by definition, **not golden.** Two runs with the same seed will not match each other byte-for-byte, and neither will match the oracle. That is the explicit, accepted trade — and it is confined to this key. **Tiers 1–2 (default and plain `SPEEDUP`) keep full parity and remain available;** nothing about this key changes how they build or what they guarantee. If you need correctness, you do not pass this key.

### 2.6.2 Mechanics

Same env-driven manifest mechanism as §2.2. `SPEEDUP_UNORDERED=1` defines **both** `SPEEDUP` and `SPEEDUP_UNORDERED` (it is a superset):

```swift
// Package.swift (top), extending §2.2
let unordered = ProcessInfo.processInfo.environment["SPEEDUP_UNORDERED"] == "1"
let speedup   = unordered || ProcessInfo.processInfo.environment["SPEEDUP"] == "1"
var defines: [SwiftSetting] = []
if speedup   { defines.append(.define("SPEEDUP")) }
if unordered { defines.append(.define("SPEEDUP_UNORDERED")) }
// …apply `defines` as swiftSettings on DuneIIWorld / DuneIISimulation.
```

Built via `SPEEDUP_UNORDERED=1 swift build -c release` (wrap in `Scripts/build-speedup.sh --unordered`). Source selects the parallel path with `#if SPEEDUP_UNORDERED … #else … #endif`, so the **sequential, ordered tick is still the code that compiles in every other build** — the parallel path is additive and isolated to this key.

### 2.6.3 What it changes in the tick (and what it must still respect)

- **RNG.** Draw order is abandoned. The clean, race-free way to get there is **per-shard (or per-thread) RNG instances** seeded deterministically from the master seed + shard index, so each worker draws from its own stream with no contention. (A single shared RNG drawn racily would be a data race on the LCG state — undefined behavior — so even here, give each worker its own generator rather than racing one.) Results still vary run-to-run because *which* entity lands on *which* shard/interleaving varies; that is the point.
- **Shared map + pools (the §1 ceiling that does *not* go away).** Concurrency here must be **memory-safe even though it need not be ordered** — no unsynchronized concurrent mutation of a Swift `Array`/pool; non-determinism is the accepted cost, corruption/crashes are not. The recommended way to get there is **lock-free**, not locked: a snapshot-read + deferred-effects fork-join where the parallel pass has *zero* shared writes and a cheap serial apply commits them. The full algorithm — and why per-object locking is the wrong tool for this dependency graph — is **§4e**; read it for the concrete design. (A `Mutex` around the few genuinely-shared records is a correct fallback if a lock-free apply proves fiddly, but it trades the context-switching cost the request wants to avoid, so treat it as a last resort.)
- **Phase order.** Team → Unit → Structure → House stay sequential as phases; each phase's entity loop parallelizes internally (§4e.3). This relaxes only within-phase entity ordering, not the real cross-phase dependencies.

### 2.6.4 Acceptance bar (it is different — see §5.2)

Because this key is non-golden, it is **forbidden** from being the build that generates or certifies any golden. Its bar is (1) **memory-safety** under the determinism stress test run as a *race detector* (TSan, K repeats — looking for crashes/corruption, not for identical hashes, which it won't have), and (2) **statistical plausibility** — over many runs, aggregate outcomes (e.g. who wins, spice mined, unit counts) stay **within OpenDUNE's own seed-to-seed spread** (core principle 1's stochastic clause), confirming we parallelized the *real* simulation and didn't silently break a rule. See §5.2.

## 3. Single-thread optimizations (do these before threading)

Pure, determinism-preserving refactors — they cannot change a golden because they don't change what is computed or in what order, only how fast:

- **Kill the copy-out/mutate/copy-in pattern.** `gameLoopUnit` repeatedly does `var u = state.units[slot]; …mutate u…; state.units[slot] = u` (e.g. `Simulation.swift:223`, `:243`, `:254`). `Unit` is a large POD value; each pair is two full struct copies. Mutate in place via direct subscript mutation, or take a scoped `&state.units[slot]` `inout`, or `state.units.withUnsafeMutableBufferPointer` around the inner loop where there is no aliasing with the same slot. Same pattern audit for structures/houses.
- **Hoist repeated lookups.** `UnitInfo[ut]`, `StructureInfo[...]`, and `adjustToGameSpeed(...)` are recomputed inside loops; compute once per entity/tick.
- **Audit per-tick allocations.** `tick()` already uses `removeAll(keepingCapacity:)` for `soundEvents`/`pendingFeedback` (good); sweep the script VM, `PoolFind`, and the path layer for transient `Array`/`Set` allocations that can be reused buffers on the runner.
- **Confirm `-O` enables the value-type/POD code to stay on the stack** (no incidental ARC on the `Sendable` POD structs).

Each lands as its own commit; the goldens stay byte-identical — that *is* the regression test.

## 4. Parallelization opportunities (ranked by safety)

§4a–§4d are **golden-preserving** — they apply to every tier including the default, and each must keep the goldens byte-identical. §4e is the **non-golden** option, available *only* under `SPEEDUP_UNORDERED` (§2.6).

### 4a. Whole independent simulations — free, highest leverage (do first)

`Simulation` is a `Sendable` value type with no shared mutable state, so **N independent sims run concurrently with zero contention.** This needs *no* change to the tick and is the easiest real parallelism we have:

- the **scenario-golden suite** and oracle comparisons (run all scenarios in parallel);
- **AI rollouts / search / parameter sweeps** (fork a snapshot, explore branches concurrently);
- batch headless runs (`duneii-headless`).

Implement with a `TaskGroup` (or the `Workflow` harness) over copies of the starting `Simulation`. This is parallelism *across* sims, orthogonal to a single tick's determinism.

### 4b. Off-golden-path subsystems — safe, per-frame wins

Work that reads a finished tick's state and never feeds back into the ordered RNG/map mutation:

- **`makeFrameInfo()`** (`FrameInfo+Make.swift`) is a pure read of `GameState` into a value snapshot — parallelize the units/structures/tiles map-over. It runs per *render frame* (host cadence), and the renderer compositing downstream of it is already a separate stage. This is the cleanest parallel win for the presentation path.
- **Explosion/animation ticks** are gated off the golden path (`tickExplosions`/`tickAnimations`); if they ever dominate the visual apps, they parallelize without parity risk.
- **Audio** is already off-thread (the `ADLMusicPlayer` producer/consumer split).

### 4c. RNG-free, entity-local sub-passes within the unit phase — modest, more work

Some `gameLoopUnit` sub-steps are RNG-free and write **only their own** `state.units[slot]`: **rotation** (`Simulation.swift:253`), **sprite animation** (`tickUnknown5`, `:281`), and **turret aim** (`:220`, reads the target's tile read-only). These could be lifted into a parallel sub-pass over the unit pool. The catch: the same loop also does RNG-drawing / shared-map work (movement, script, tile re-claim) that must stay sequential and in order. So this means **splitting the unit loop into a parallel RNG-free pass + a sequential RNG/map pass**, which adds real complexity. Only pursue if profiling shows these sub-steps are hot; verify with the §5 identical-hash stress test.

### 4d. Parallel pathfinding into a cache — advanced, careful

The `Pathfinder` is a **pure, RNG-free** function of (map snapshot, start, goal) (confirmed: no RNG draws), so multiple path computations are embarrassingly parallel *in isolation*. The obstacle is that path **requests** are produced by the sequential, RNG-drawing script logic — you don't know which units will request a path until you've run the ordered logic, so you can't trivially batch them ahead. Tractable approaches, only if pathfinding profiles as the dominant cost: a cross-tick **path cache** keyed on (start, goal, map-version), or precomputing paths for units whose requests are state-determined (not RNG-gated). High risk to determinism; gate behind the stress test and treat as last.

### 4e. Full intra-tick parallelism — `SPEEDUP_UNORDERED` only, non-golden

This is the bucket §1 closes for parity builds and §2.6 reopens by giving up the RNG-order contract. **Under `SPEEDUP_UNORDERED` only**, the per-phase entity loops (the whole of `gameLoopUnit`/`gameLoopStructure`/etc., *including* their RNG-drawing and shared-map sub-steps) run concurrently. Its payoff and fences in brief:

- **Why it's only here:** it reorders RNG draws and interleaves shared writes by design, so it cannot keep the goldens — hence the dedicated, explicitly non-golden key, while §4a–§4d (and the whole default/`SPEEDUP` experience) keep parity.
- **What it is *not*:** a license to race memory. The §1 "shared map + pools" ceiling still binds — unsynchronized concurrent `Array`/pool mutation is UB, not "acceptable non-determinism."
- **Payoff:** the only option that lifts the §1 *single-tick* ceiling — it scales one sim's tick across cores, complementary to §4a (many sims in parallel). For rollouts/sweeps §4a is usually the better first lever; reach for §4e only when one sim's tick itself must be faster and parity is genuinely not required.

The rest of §4e is the **algorithm** the user asked for. Read §4e.1 (the dependency answer) before §4e.2 (the design), because the design exists *because of* the dependencies.

#### 4e.1 Are there dependent parts? Yes — pervasively, and they can't be statically partitioned

The honest answer to "can we split game logic into distinct pieces whose data flows never intersect?" is **no, not by a static partition** — and it's worth being precise about why, because it dictates the design.

The tick is four sequential phases (Team → Unit → Structure → House), each a `PoolFind` loop over entities (`Simulation.swift:113`). Within a phase, an entity's work reads and writes **state that belongs to other entities**, and *which* others is decided at runtime, not statically:

- **Combat / damage (the example from the request, confirmed in code).** A unit's impact writes a *different* unit's `hitpoints` (`UnitImpact.swift:33`), and an explosion's area damage **loops over every unit within a blast radius and writes each one's health** (`UnitImpact.swift:151`+). So your "rocket position → same-tick explosion → target unit's health changes" is exactly real: one entity fans out writes onto an arbitrary, *position-determined* set of other entities. No assignment of units to threads can keep combat inside a thread, because any unit can damage any other.
- **Map claim/release.** Ground units release/re-claim their tile on the shared `state.map` (`Simulation.swift:273`, `unitUpdateMap`); movement writes the from- and to-tiles. Neighbours touch the same or adjacent cells → spatially-local write conflicts.
- **Shared economy.** Many structures/units of one house all read-modify-write the *same* `state.houses[h]` record (credits, power, harvesters-incoming) in the structure and house phases.
- **Spawning.** Combat/economy allocate new entities from the shared pools (`unitCreate`, the starport frigate `Simulation.swift:554`) — a write to a shared allocator, not to a private slot.
- **Cross-phase order.** House economy (`gameLoopHouse`) depends on what Structure produced this tick; Structure on what Unit produced. The phase order carries real dependencies.

So the dependency graph is **dense and data-determined**. You cannot pre-slice "rockets here, infantry there, economy over there" such that the slices never touch each other — the touch set only exists *after* you run the logic. That rules out the naive "split by subsystem, run each on a thread" and it rules out per-object locking (the locks would be on a hot, contended, runtime-chosen set — exactly the context-switching cost you want to avoid).

#### 4e.2 The design: snapshot-read + deferred-effects fork-join (lock-free)

The way to parallelize a dense, data-determined dependency graph **without locks** is not to partition the *data* but to separate the two *phases of access*: a parallel **read-compute** pass that touches no shared writable state, then a cheap **apply** pass that commits the writes. This is the standard double-buffer + command-buffer pattern; it fits `SPEEDUP_UNORDERED` perfectly because the one behavioural change it introduces — every entity reads start-of-phase state instead of the partially-updated mid-phase state the sequential loop sees — is precisely the kind of bounded, ≤1-phase-stale approximation this non-golden key exists to allow.

Per phase (keep the four phases sequential — see §4e.3):

1. **Freeze** the phase's input as an immutable snapshot. `GameState`'s pools are COW value types, so freezing is a cheap retain, not a deep copy — workers read the snapshot, writers build a fresh output buffer.
2. **Fork — parallel, lock-free.** Shard the entity pool into chunks (by slot range; ~64 entities/chunk to amortise scheduling, not one task per entity). Run the chunks with `DispatchQueue.concurrentPerform` (a parallel-for with a built-in join barrier and no per-task allocation — this *is* the "do a part, then signal done" the request describes, with the signal being the barrier). Each worker, for each of its entities:
   - **reads** the frozen snapshot freely (self + any other entity/tile) — reads of immutable data never conflict, so **no locks**;
   - **writes its own** next-state into `newBuffer[slot]` — each slot has exactly one writer, so **no conflict**;
   - draws RNG from a **per-worker** generator seeded `master ⊕ shardIndex` (§2.6.3) — no shared RNG state;
   - **defers every cross-entity write** — damage, spawn, map-claim, spice/credit delta, sound — by appending a small value to its **thread-local effect buffer** instead of mutating shared state.
3. **Join** (the `concurrentPerform` barrier).
4. **Apply — the only serial part, and deliberately small (O(#effects), not O(#entities × work)).** Drain the per-worker effect buffers into the real state:
   - **Economy** folds first and is even *order-independent*: sum each worker's per-house credit/power **delta accumulators** into `state.houses` (addition commutes → this part is actually deterministic).
   - **Damage** groups by target slot and applies (sum then clamp; resolve deaths after). Order across sources within a tick may vary — accepted under UNORDERED.
   - **Map claims** resolve per cell with a fixed tiebreak (lowest slot wins) so a contested tile is at least consistent.
   - **Spawns / sounds / feedback** concatenate; spawns allocate real pool slots here (allocation stays single-threaded, so the pool itself is never raced).
   If the apply ever profiles as the bottleneck, shard *it* by target-slot range so each apply-worker owns a disjoint set of targets — still lock-free.
5. **Commit** `newBuffer` as the phase's output; it becomes the next phase's frozen input.

This answers all four points in the request: work **is** split into parts run on separate threads that signal completion (the barrier); the **dependent parts are handled** not by forbidding them but by routing every cross-entity write through the deferred-effects channel so the parallel pass has *zero* shared writes; and it is **lock-free** — the only coordination is the join barrier and a commutative reduce, never a per-object lock.

#### 4e.3 Granularity, thresholds, and what stays sequential

Parallelism has overhead; spend it only where the work is:

- **Per-phase, not per-tick.** Snapshot/fork/join **inside** each phase, keeping the four phases ordered. This relaxes only *within-phase entity ordering* (cheap, the §4e.2 staleness) while preserving the real *cross-phase* dependencies, and avoids four overlapping mutation streams.
- **Parallelise the Unit phase first; it dominates** (most entities, movement + script + combat). The **Team phase stays sequential** (few teams, the script aborts the rest on a halt — `Simulation.swift:398`); the **House phase stays sequential** (a handful of houses), though its economy is the natural sink for the unit/structure delta accumulators. Structure phase is a middle case — parallelise if it profiles hot.
- **Entity-count threshold.** Below ~128 active entities the fork/join overhead exceeds the gain, so fall back to the existing sequential tick. This guard is what stops `SPEEDUP_UNORDERED` from being *slower* than the plain build on small scenarios — directly addressing the "don't add context-switching that slows everything down" concern.
- **Pool size = `min(cores, ...)`, chunked.** Never one task per entity; size the worker pool to the core count and stride the pool into chunks, so there's no oversubscription and effect buffers stay per-core (no false sharing).

#### 4e.4 The main engineering cost (be honest about it)

The parallel pass is only lock-free because **every cross-entity mutation is rerouted through the effect channel**. Today those writes are applied inline (e.g. `UnitImpact` writes target hitpoints directly). Converting them to deferred effects — an `enum Effect { case damage(target, amount) … }`, thread-local buffers, and the apply/reduce — is the real work, and it must be done for *every* inline cross-entity write in the parallelised phase or that write reintroduces a race. This is why §4e sits late in the rollout (§6 phase 5) and behind its own key: it is a substantial, isolated refactor, not a flag flip. Its correctness bar is §5.2 (TSan-clean + within-spread), never the goldens.

#### 4e.5 Alternative framing — the work-queue / continuation model (and why a *wavefront* is the version that pays off)

A natural alternative is to **decompose each cascade into fine steps on a queue**: process the rocket; if it impacts, enqueue an "explosion" step; continue draining; when the explosion step runs, it enqueues one "apply damage" step per affected unit; workers pull steps off the queue in parallel. This is a real, well-known model — it is **task-graph / dataflow scheduling**, and its per-entity-serialized form is the **actor model** (one mailbox per entity). It is worth being precise about what it does and does not buy, because the intuition "split into basic steps ⇒ parallelism" is half-right and half a trap.

**What it does *not* solve — the conflict, which is the whole problem.** Decomposing into steps does **not** make conflicting writes safe. Two "apply damage to unit B" steps (from two explosions) can sit in the queue at once; run them on two workers and they race on B's hitpoints exactly as before. So a step queue still needs one of the same three answers from §4e.1:
- **one step per entity at a time** — i.e. an actor/mailbox per entity, serializing all steps that touch it. But that is *precisely the per-object synchronization you wanted to avoid*, now paid as mailbox enqueue/dequeue + cache-missing message passing on every interaction — generally *more* overhead than the fork-join's plain array writes, not less; or
- **deferred effects** — steps don't write shared state, they emit effects applied later. But that is already §4e's mechanism; the queue adds nothing to the safety story, only a different scheduler in front of it; or
- **locks** — rejected for the same reason as everywhere else.

So the decomposition relocates the conflict into the scheduler; it does not dissolve it.

**Where its parallelism actually comes from (and where it doesn't).** The chain rocket → explosion → damage is a **dependency chain**: each link needs the previous link's result, so it is *inherently serial along its length* — the queue cannot parallelize *within* one cascade. The parallelism is the same source as fork-join: **many independent items in flight at once** (many rockets, many units). The one genuine extra the dataflow model offers is **pipeline overlap** — explosion-of-rocket-1 can compute while rocket-2 still moves and damage-from-rocket-0 still applies, three stages of three chains running with no barrier between stages, whereas per-phase fork-join inserts a barrier between "all moves / all explosions / all damage." That saves the barrier-stall idle time — a real but usually *small* win, because per-tick stage counts are low and the barrier is cheap relative to the model's costs below.

**The two costs — which are exactly your stated fears.** (1) **Queue contention:** a single concurrent queue that N workers push/pop is an atomic-CAS hotspot hit thousands of times per tick — "locking that slows everything down," just moved into the scheduler. Per-worker work-stealing deques fix it but are hard to make correct and lock-free. (2) **Fine-grain overhead:** a unit move is *cheap arithmetic*; wrapping each basic step as a separately-scheduled queue item means the enqueue/dequeue/dispatch (and any closure allocation) routinely costs more than the step itself. Fine-grained tasking is the classic way to make a parallel program *slower* than a tight loop. The fork-join model dodges both by chunking ~64 entities per task; the moment you chunk the queue to fix overhead, you are converging back toward fork-join anyway.

**The synthesis that keeps the good part — drain the queue in barriered *waves* (bulk-synchronous), chunked.** The cascade expressiveness of your idea is real and the static four-phase fork-join handles irregular, data-dependent cascade *depth* awkwardly. The fix is to keep the queue but **not** run it as a free-for-all concurrent queue: run it as a **wavefront**.
- A "wave" is the current queue contents. Process the whole wave as a **chunked fork-join** (§4e.2 mechanics: read snapshot, write own slot, defer cross-entity writes as effects). Steps produced this wave go into the *next* wave's queue, not back into the live one.
- Apply the wave's effects (the cheap serial/sharded reduce), then start the next wave from the queue the wave just filled. Repeat until the queue is empty.
- Wave 0 = "advance all units"; it emits explosion steps. Wave 1 = "all explosions"; emits damage steps. Wave 2 = "all damage"; may emit death/secondary-explosion steps. Wave 3 = … Cascades of arbitrary depth terminate naturally when a wave produces no successors — and termination is **trivial to detect** (empty queue after the barrier), avoiding the distributed-quiescence problem a free-running concurrent queue has.

This is the **bulk-synchronous parallel (BSP)** discipline. It *is* your queue/continuation idea — "split complex processing into basic steps, enqueue follow-ups, drain in parallel" — but drained in barriered, chunked waves instead of as a contended concurrent queue. That keeps the dynamic-cascade expressiveness while killing the contention, the fine-grain overhead, and the termination-detection complexity. It also stays lock-free via the same deferred-effects channel (§4e.2/§4e.4) — the wave model is a *scheduler choice layered on that channel*, not a replacement for it.

**Verdict.** Will it work? Yes — as a wavefront/BSP layered on deferred effects. As a *free-running concurrent step queue with per-entity actors*, it also "works" but typically runs **slower** than chunked fork-join for this workload, because the per-step and queue-contention costs dominate cheap entity work — i.e. it spends exactly the context-switching/locking budget you set out to avoid. Recommendation: treat §4e.2's chunked fork-join as the default for the regular high-count phase, and adopt the §4e.5 wavefront only for the irregular cascade tail (explosions/deaths/secondary blasts) if profiling shows that tail is where the time goes. Same correctness bar either way: §5.2, never the goldens.

## 5. Verification & acceptance (mandatory for every change)

### 5.1 Parity builds (default + `SPEEDUP`) — the golden bar

1. **Goldens byte-identical** (non-SPEEDUP): `Scripts/check.sh` green, and `Scripts/gen-scenario-goldens.sh` produces no diff. This catches any reordered/dropped RNG draw or altered logic.
2. **SPEEDUP equivalence golden** (§2.5): final-state/`SaveGame`-hash identical between a normal release build and a SPEEDUP build, over a fixed long scenario.
3. **Determinism stress test for *any* parallel change:** run the same scenario K times (and across thread counts) and assert an identical final-state hash every time. A data race that reorders or corrupts shows up here as a nondeterministic hash; without this test a parallel bug is invisible until a golden happens to catch it. **No parallel change merges without it.**
4. **Benchmark harness:** a fixed headless scenario run for a fixed tick count, timed, reporting ticks/sec (add `Scripts/bench.sh`; record the baseline before optimizing so each step's gain is measured, not assumed).

### 5.2 `SPEEDUP_UNORDERED` — the non-golden bar (different on purpose)

This tier **cannot** be held to byte-identity (§2.6.4) — its acceptance is:

1. **Memory-safety, not hash-identity.** Run the §5.1 stress harness (item 3) under **ThreadSanitizer**, K repeats across thread counts, as a *race detector*: the pass condition is "no TSan report, no crash, no corruption." Do **not** assert an identical final-state hash — by design there isn't one. This is the gate that distinguishes "accepted non-determinism" from "undefined behavior."
2. **Statistical plausibility (within OpenDUNE's spread).** Over many seeds/runs, aggregate outcomes (winner, spice mined, surviving unit/structure counts, game length) stay **within OpenDUNE's own seed-to-seed spread** (core principle 1's stochastic clause). This confirms the parallelized tick is still the *real* simulation and didn't silently drop or double a rule under the relaxed ordering.
3. **Never certifies a golden.** A `SPEEDUP_UNORDERED` binary must not be the one that generates/regenerates any scenario or trace golden, and CI must keep generating goldens from a parity build (§5.1). Keep the two paths from being confused — e.g. have the unordered build stamp its run output as "non-golden."

## 6. Phased rollout

1. **SPEEDUP flag + strip tracing + release build + `Scripts/bench.sh` + the equivalence golden.** (Requirements 1 & 2; immediate, low-risk, measurable.)
2. **Single-thread §3 refactors** (in-place mutation, hoisting, allocation sweep), each verified by unchanged goldens + a bench delta.
3. **§4a multi-sim parallelism** (golden suite / tooling) and **§4b `makeFrameInfo` parallel build.**
4. **(Optional, profile-gated) §4c intra-phase RNG-free sub-passes**, then **§4d path cache** — each behind the §5.1 stress test.
5. **(Optional, separate track) `SPEEDUP_UNORDERED` + §4e full intra-tick parallelism** (§2.6). Independent of phases 1–4 and of the golden bar: a dedicated key for the throughput-only use cases, accepted against the §5.2 (non-golden) bar — TSan-clean + within-spread. Build it only when a throughput use case (rollouts/sweeps) actually needs single-sim ticks faster than §4a delivers; never let it regress or replace the parity builds.

## 7. Risks

- **Silent determinism breakage from a race** — for the *parity* builds the *only* defense is the §5.1 identical-hash stress test; it is mandatory, not optional, for §4c/§4d.
- **`#if !SPEEDUP` drift** — the guarded blocks must stay *observation-only*; never let logic or an RNG draw live inside one, or SPEEDUP and normal builds would diverge (and the §2.5 equivalence golden would — correctly — fail).
- **`SPEEDUP_UNORDERED` leaking into a parity build or a golden** (new, this revision) — the biggest risk of the new key. It must be (a) impossible to enable accidentally (distinct env var; never implied by `SPEEDUP`), (b) never the binary that generates a golden (§5.2.3), and (c) clearly stamped as non-golden in its output so a sped-up run is never mistaken for a faithful one. Confine the `#if SPEEDUP_UNORDERED` parallel path to additive code so the sequential ordered tick is still what every other build compiles.
- **A *data race* (not just non-determinism) in `SPEEDUP_UNORDERED`** — relaxing RNG order is fine; racing a shared `Array`/pool is undefined behavior. The §5.2 TSan gate is mandatory; per-shard RNG + sharded/locked shared writes (§2.6.3) are the required shape, not an optimization.
- **Over-investing in tick threading** — the determinism ceiling (§1) means single-sim *parity* tick parallelism has a low payoff ceiling; §2/§3/§4a are where the parity wins are. Don't build §4c/§4d without a profile proving the need, and reach for §4e (`SPEEDUP_UNORDERED`) only when a throughput use case truly needs a single sim's tick faster than §4a's many-sim parallelism delivers.

## 8. Measured results — first experiment (2026-06-02)

The user asked the direct question: build a busy scenario, run it sequentially, then run it parallel — which is faster? We built the harness and measured. **Answer: sequential, decisively. Naive parallelism is net-negative on this engine today, and the reason is instructive — it is not a flaw in the parallel harness.**

### 8.1 What was built (branch `parallelization-experiment`)

- **`Apps/simbench`** — a benchmark. Generates a deliberately busy synthetic scenario on a rock map (per "group": refinery + heavy factory building a tank + construction yard building a starport + 2 windtraps + a harvester harvesting + a cross-house quad duel), then times five things: `[0]` a pure-compute parallel **sanity** check, `[1]` the sequential tick with a per-phase breakdown, `[2]` the parallel unit phase end-to-end, `[3]` an isolated **divergence-free** unit-phase A/B (sequential vs parallel unit phase on the *same* per-tick input), and `[4]` N independent whole sims in parallel (§4a).
- **`Frameworks/DuneIISimulation/Simulation+Parallel.swift`** — a **non-golden** parallel unit phase (`gameLoopUnitParallel`): snapshot the `Simulation` (COW), shard the live unit slots, run each shard on its own value copy via `DispatchQueue.concurrentPerform`, merge each shard's own slots back. Cross-shard writes (combat damage to another shard's unit, spawns, map claims, credit deltas) are dropped — the accepted §4e approximation for a *throughput* measurement. Plus `tickParallel`, `tickTimed`/`PhaseTimings`. `gameLoopUnit` was refactored into a slot-parameterised `runUnitPhase` (behaviour-preserving: the goldens stayed byte-identical).
- **Pool escape hatch** — `Pool.unitIndexMax` reads `DUNEII_UNIT_POOL` (default 102 ⇒ suite unchanged). The faithful pool is **102 unit slots partitioned by type** (all ground combat units share `indexStart..indexEnd` = 22..101 ⇒ ~80 ground units max), so a faithful single sim can never hold a "massive" army or reach the ~128-entity threshold (§4e.3) where intra-tick parallelism could pay. The benchmark injects cloned units past the cap (`benchInjectUnit`) to probe scale.

### 8.2 Numbers (16-core Apple Silicon, `-c release`)

| Scenario | Sequential tick | `[0]` pure-compute parallel | `[2]`/`[3]` parallel **unit phase** | `[4]` 16 **independent** sims parallel |
|---|---|---|---|---|
| Faithful (56 units, 79 structures) | **0.008 ms/tick** (~123k ticks/s); unit phase 75%, structure 22% | **12.3× faster** | ~neutral (falls back below the 64-unit threshold) | **0.10× — i.e. ~10× *slower*** |
| Massive (2000 units, pool 8000) | **0.40 ms/tick** (~2500 ticks/s); unit phase 99.5% | **10.1× faster** | **0.11× — ~9× *slower*** ([2] snapshot path: ~78× slower) | **0.14× — ~7× *slower*** |

### 8.3 Diagnosis (this is the valuable part)

Pure compute parallelizes 10–12× on the same machine, harness, and core count — so the slowdown is **intrinsic to the simulation workload**, not the test. The cause was narrowed by two controlled isolation runs (`simbench` `[5]`/`[6]`):

- **`[5]` — de-shared `ScriptInfo`.** Gave each parallel worker its *own* deep copy of the script bytecode buffers (no shared COW). **No change** (still 0.11×/0.13×) ⇒ the contention is **not** the script buffers.
- **`[6]` — nothing shared but the global tables + allocator.** Each worker builds *and* ticks a fully independent sim with de-shared assets. **Still ~0.15–0.27×** (4–7× slower) ⇒ the bottleneck is what remains shared/global, not anything per-sim.

So the real causes are:

1. **The sim is memory-/allocation-bound, not compute-bound.** Pure register-bound compute scales 12×; the moment work streams the large per-sim arrays (units, the 64×64 map, `mapBaseTileID`) and allocates transient `Array`/`Set` per unit (script VM, pathfinder, `PoolFind`, targeting), 16 cores saturate memory bandwidth and serialize on the allocator's locks/magazines. This is the dominant effect and it caps *all* threading.
2. **Per-tick stat-table lookups copy reference-counted data.** Every unit-tick does `UnitInfo[ut]` / `StructureInfo[…]` etc. Those table structs embed ref-counted members (`String` names, nested arrays), so each lookup retains/releases *shared* storage — 16 cores ping-ponging the same refcount cache lines. (De-sharing `ScriptInfo` didn't help precisely because the hot shared data is the *stat tables*, not the script.)
3. **Snapshot-per-shard COW tax (intra-tick only).** `gameLoopUnitParallel` copies the whole units/map arrays per shard per tick on first write → ~78× slower at 2000 units. The §4e.2 design (single shared output buffer + deferred effects) avoids this, but it is moot until 1–2 are fixed.

### 8.4 What the measurement says to do (reaffirms §3/§6, now with evidence)

**Do not thread anything yet.** The ordering is now empirical, not theoretical:

1. **§3 single-thread first, and it is a *prerequisite*, not an option — now with a measured target.** (a) Eliminate per-tick transient allocations (reuse buffers on the runner; the sim is allocation-bound). (b) Make the hot stat-table lookups **non-retaining**: `UnitInfo[ut]`/`StructureInfo[…]` copy structs that embed `String`/nested-array members, so every per-tick lookup does atomic retain/release on shared storage — the `[5]`/`[6]` runs proved this (not the script buffers) is the shared hot data. Fixes: read the few scalar fields the hot path needs without copying the whole struct, hold the tables via `UnsafeBufferPointer`/`borrowing`, or strip ref-counted members out of the hot-path struct. Until (a)+(b) are done, *every* threading scheme (even §4a) loses — and it loses to **memory bandwidth**, which also caps the eventual ceiling.
2. **§2 SPEEDUP build** — the `-O`/strip win, independent of threading.
3. **§4a (many independent sims)** is the highest-leverage parallelism *in principle*, but today it is ~7–10× **slower** because of cause 1. It only becomes the easy win the plan claims **after** step 1 removes the shared-refcount contention. Re-measure then.
4. **§4e intra-tick** stays last and behind `SPEEDUP_UNORDERED`; the measurement-grade snapshot impl here is a worst case kept only for the A/B. A real attempt needs the deferred-effects single-buffer design — and is only worth building if a throughput use case still needs it after steps 1–3.
5. **"Massive units" is a separate, non-faithful axis.** The faithful 102-slot, per-type-partitioned pool caps a real sim at ~80 ground units — below any threshold where intra-tick threading helps. Scaling to thousands requires raising the caps (the `DUNEII_UNIT_POOL` hatch), which is by definition not a faithful build.

Bottom line for the original question: **the sequential tick is already very fast (123k ticks/s at faithful scale, 2500 ticks/s at 2000 units), and is faster than every parallel variant measured.** The lever for "larger workloads" is (a) kill allocations + shared-refcount reads, then (b) run many independent sims in parallel — not threading one tick.
