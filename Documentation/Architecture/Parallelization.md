# Tick Parallelization & Optimization — plan

**Status: design only. Not implemented.** This is the plan for making `Simulation.tick()` faster, in two tracks the user asked for: (1) a separate **`SPEEDUP` compilation mode** (a build parameter) that (2) removes all non-essential debug output, plus the broader parallelization/optimization analysis that mode enables. Read `Architecture/Overview.md` and `Architecture/ParityHarness.md` first; this assumes the four-phase tick (`Frameworks/DuneIISimulation/Simulation.swift`).

## 0. The non-negotiable invariant (read before anything else)

**Determinism + behavioral parity are the acceptance bar for *every* change here** (core principles 1 and 5). Same scenario + seed + command stream ⇒ byte-identical run; and the run stays bit-identical to the OpenDUNE oracle wherever behavior is deterministic. Concretely, two things must stay byte-identical after any optimization or parallelization:

- **the scenario goldens** (`ScenarioGoldenTests`, regenerated by `Scripts/gen-scenario-goldens.sh`), and
- **the RNG draw-stream golden** — our per-tick `Random256`/`RandomLCG` draws, in number and order, match the oracle's (see `RngTraceSink` + insight `sim-rng-stream-unpinned-wobble`).

This is not a guideline, it is the test. Any change that flips a golden byte is wrong **by definition** — it has reordered or dropped an RNG draw, raced a shared mutation, or altered logic. The goldens are the safety net that makes aggressive optimization safe to attempt.

## 1. The determinism ceiling on parallelism

The ordered tick is *mostly inherently sequential*, and it is important to be honest about why, so we don't chase parallelism that can't preserve parity:

- **One shared RNG stream, order-significant.** `Random256` and `RandomLCG` are drawn in a single global order across all four phases and all entities. The oracle draws the *same number in the same order*; that order is the parity contract. Any scheme that lets two entities draw concurrently reorders the stream and diverges. So **every code path that draws RNG must run in the existing sequential order** — this rules out naive "parallel-for over all units."
- **One shared map + shared pools.** Per-entity work writes the map (`unitUpdateMap`, spice, overlays) and reads/writes other entities (targets, houses, the object pools). Concurrent writers race.
- **Fixed phase order** Team → Unit → Structure → House, each iterating a deterministic `PoolFind` cursor.

What *is* independent, and therefore parallelizable, falls into three buckets (§4): work that draws no RNG and touches only its own entity; subsystems that are entirely off the golden/oracle path; and *whole separate simulations*. The single biggest single-sim speedups, though, come from **§2 (SPEEDUP) and §3 (single-thread)**, not from threading one tick.

## 2. The `SPEEDUP` compilation mode (requirements 1 + 2)

### 2.1 What it is

A compile-time configuration that produces a faster binary by (a) **stripping all observation/debug instrumentation** to nothing and (b) building optimized. It is selected by a build parameter, never at runtime, so the stripped code has *zero* footprint (not even a branch).

### 2.2 Mechanics — how the flag is passed

Two viable mechanisms; we recommend the second for a first-class named build:

- **Ad-hoc:** `swift build -c release -Xswiftc -DSPEEDUP`. `-D SPEEDUP` defines the `SPEEDUP` compilation condition for every Swift target; `-c release` gives `-O`. No manifest change. Good for a quick A/B.
- **Recommended — env-driven manifest.** `Package.swift` reads an env var and conditionally injects the define (and any extra flags) into the engine targets, so a single documented command builds the mode:

  ```swift
  // Package.swift (top)
  import Foundation
  let speedup = ProcessInfo.processInfo.environment["SPEEDUP"] == "1"
  let speedupSettings: [SwiftSetting] = speedup ? [.define("SPEEDUP")] : []
  // …then on DuneIIWorld / DuneIISimulation (and any target with hot-path tracing):
  //   swiftSettings: speedupSettings
  ```

  Built with `SPEEDUP=1 swift build -c release` (wrap it in `Scripts/build-speedup.sh`, alongside the repo's `TMPDIR`/`xcrun`/`--disable-sandbox` quirks). The env-var form keeps the manifest the single source of truth and lets SPEEDUP also carry optimizer flags (§2.4) without retyping `-Xswiftc`.

The source uses `#if !SPEEDUP` to keep the instrumentation in normal builds and compile it out under SPEEDUP.

### 2.3 What it strips (requirement 2 — "remove all non-essential debug output")

The simulation has **no `print`/`Logger`/`os_log`/`#if DEBUG` in its hot path** (audited). "Debug output" here is precisely the **parity/trace instrumentation**, all currently runtime-gated by a `nil` optional (cheap, but not free, and a maintenance surface). SPEEDUP removes the hooks *and* the fields they test:

- **RNG trace hooks.** `Random256.next()` (`Rng/Random256.swift:39`) and `RandomLCG.range()` (`Rng/RandomLCG.swift:44`) end with `traceSink?.recordR256/LCG(...)`. Under SPEEDUP, wrap the `public var traceSink: RngTraceSink?` property *and* the call in `#if !SPEEDUP`, so each draw is pure arithmetic with no optional load/branch. (These draws happen thousands of times per tick — this is the one hot-loop hook worth removing.)
- **Script decision-trace.** The per-opcode `if let tracer … ScriptTraceLine.decode … tracer.record` in `StructureScriptRunner.run` (`Script/Natives/StructureScriptRunner.swift:73`) — and the equivalent unit/team paths if/when added — wrap in `#if !SPEEDUP`. `ScriptTraceLine.decode` re-decodes every opcode for the trace; gone under SPEEDUP.
- **Any future inline trace `Log`.** Standing memory rule is "every sim mutation gets a trace `Log`." When such a logging seam is added, route it through a single `@inline(__always)` shim that is `#if !SPEEDUP` (a no-op body under SPEEDUP), so the rule and the speed mode don't fight.

The EMC disassembler, `assetgen`, and the test/oracle harnesses are **not** in any shipped tick — they are unaffected and need no guarding.

### 2.4 Optimizer settings SPEEDUP may carry

- **`-O` (release): mandatory.** This is the single largest win. (For calibration: the OPL3 synth measured ~1.9× real-time in debug vs ~0.17× in release — a ~11× swing; the sim is the same kind of arithmetic-heavy Swift and benefits similarly.) SPEEDUP should always be built `-c release`.
- **`-Ounchecked`: opt-in, later, with care.** It removes bounds and overflow traps. The sim already uses explicit wrapping (`&+`/`&-`) where wraparound is intended, so traps mostly guard *pool-index* and *array* access. Removing them risks turning a real porting bug into silent corruption. Defer until the goldens + the §5 stress test give confidence; gate it behind a *second* env var (`SPEEDUP_UNCHECKED=1`) so plain SPEEDUP stays safe.

### 2.5 The SPEEDUP correctness rule + its golden

**SPEEDUP must be behaviorally identical to a normal build** — it may only *remove observation*, never change logic, math, or RNG order. The acceptance test is a **flag-on/flag-off equivalence golden**: run a fixed scenario for N ticks in both a normal release build and a SPEEDUP build, and assert the final `GameState` (or its serialized `SaveGame` hash) is byte-identical. Because SPEEDUP strips the trace sinks, the *trace-based* goldens (RNG-stream, script decision-trace) necessarily run in **non-SPEEDUP** builds — SPEEDUP is verified by the final-state equivalence, the trace goldens verify the logic, and the two together cover it.

## 3. Single-thread optimizations (do these before threading)

Pure, determinism-preserving refactors — they cannot change a golden because they don't change what is computed or in what order, only how fast:

- **Kill the copy-out/mutate/copy-in pattern.** `gameLoopUnit` repeatedly does `var u = state.units[slot]; …mutate u…; state.units[slot] = u` (e.g. `Simulation.swift:223`, `:243`, `:254`). `Unit` is a large POD value; each pair is two full struct copies. Mutate in place via direct subscript mutation, or take a scoped `&state.units[slot]` `inout`, or `state.units.withUnsafeMutableBufferPointer` around the inner loop where there is no aliasing with the same slot. Same pattern audit for structures/houses.
- **Hoist repeated lookups.** `UnitInfo[ut]`, `StructureInfo[...]`, and `adjustToGameSpeed(...)` are recomputed inside loops; compute once per entity/tick.
- **Audit per-tick allocations.** `tick()` already uses `removeAll(keepingCapacity:)` for `soundEvents`/`pendingFeedback` (good); sweep the script VM, `PoolFind`, and the path layer for transient `Array`/`Set` allocations that can be reused buffers on the runner.
- **Confirm `-O` enables the value-type/POD code to stay on the stack** (no incidental ARC on the `Sendable` POD structs).

Each lands as its own commit; the goldens stay byte-identical — that *is* the regression test.

## 4. Parallelization opportunities (ranked by safety)

### 4a. Whole independent simulations — free, highest leverage (do first)

`Simulation` is a `Sendable` value type with no shared mutable state, so **N independent sims run concurrently with zero contention.** This needs *no* change to the tick and is the easiest real parallelism we have:

- the **scenario-golden suite** and oracle comparisons (run all scenarios in parallel);
- **AI rollouts / search / parameter sweeps** (fork a snapshot, explore branches concurrently);
- batch headless runs (`duneii-headless`).

Implement with a `TaskGroup` (or the `Workflow` harness) over copies of the starting `Simulation`. This is parallelism *across* sims, orthogonal to a single tick's determinism.

### 4b. Off-golden-path subsystems — safe, per-frame wins

Work that reads a finished tick's state and never feeds back into the ordered RNG/map mutation:

- **`makeFrameInfo()`** (`FrameInfo+Make.swift`) is a pure read of `GameState` into a value snapshot — parallelize the units/structures/tiles map-over. It runs per *render frame* (host cadence), and the renderer compositing downstream of it is already a separate stage. This is the cleanest parallel win for the presentation path.
- **Explosion/animation ticks** are gated off the golden path (`tickExplosions`/`tickAnimations`); if they ever dominate the visual apps, they parallelize without parity risk.
- **Audio** is already off-thread (the `ADLMusicPlayer` producer/consumer split).

### 4c. RNG-free, entity-local sub-passes within the unit phase — modest, more work

Some `gameLoopUnit` sub-steps are RNG-free and write **only their own** `state.units[slot]`: **rotation** (`Simulation.swift:253`), **sprite animation** (`tickUnknown5`, `:281`), and **turret aim** (`:220`, reads the target's tile read-only). These could be lifted into a parallel sub-pass over the unit pool. The catch: the same loop also does RNG-drawing / shared-map work (movement, script, tile re-claim) that must stay sequential and in order. So this means **splitting the unit loop into a parallel RNG-free pass + a sequential RNG/map pass**, which adds real complexity. Only pursue if profiling shows these sub-steps are hot; verify with the §5 identical-hash stress test.

### 4d. Parallel pathfinding into a cache — advanced, careful

The `Pathfinder` is a **pure, RNG-free** function of (map snapshot, start, goal) (confirmed: no RNG draws), so multiple path computations are embarrassingly parallel *in isolation*. The obstacle is that path **requests** are produced by the sequential, RNG-drawing script logic — you don't know which units will request a path until you've run the ordered logic, so you can't trivially batch them ahead. Tractable approaches, only if pathfinding profiles as the dominant cost: a cross-tick **path cache** keyed on (start, goal, map-version), or precomputing paths for units whose requests are state-determined (not RNG-gated). High risk to determinism; gate behind the stress test and treat as last.

## 5. Verification & acceptance (mandatory for every change)

1. **Goldens byte-identical** (non-SPEEDUP): `Scripts/check.sh` green, and `Scripts/gen-scenario-goldens.sh` produces no diff. This catches any reordered/dropped RNG draw or altered logic.
2. **SPEEDUP equivalence golden** (§2.5): final-state/`SaveGame`-hash identical between a normal release build and a SPEEDUP build, over a fixed long scenario.
3. **Determinism stress test for *any* parallel change:** run the same scenario K times (and across thread counts) and assert an identical final-state hash every time. A data race that reorders or corrupts shows up here as a nondeterministic hash; without this test a parallel bug is invisible until a golden happens to catch it. **No parallel change merges without it.**
4. **Benchmark harness:** a fixed headless scenario run for a fixed tick count, timed, reporting ticks/sec (add `Scripts/bench.sh`; record the baseline before optimizing so each step's gain is measured, not assumed).

## 6. Phased rollout

1. **SPEEDUP flag + strip tracing + release build + `Scripts/bench.sh` + the equivalence golden.** (Requirements 1 & 2; immediate, low-risk, measurable.)
2. **Single-thread §3 refactors** (in-place mutation, hoisting, allocation sweep), each verified by unchanged goldens + a bench delta.
3. **§4a multi-sim parallelism** (golden suite / tooling) and **§4b `makeFrameInfo` parallel build.**
4. **(Optional, profile-gated) §4c intra-phase RNG-free sub-passes**, then **§4d path cache** — each behind the §5 stress test.

## 7. Risks

- **`-Ounchecked` hiding real bugs** (§2.4) — keep it opt-in and behind its own flag.
- **Silent determinism breakage from a race** — the *only* defense is the §5 identical-hash stress test; it is mandatory, not optional, for §4c/§4d.
- **`#if !SPEEDUP` drift** — the guarded blocks must stay *observation-only*; never let logic or an RNG draw live inside one, or SPEEDUP and normal builds would diverge (and the §2.5 equivalence golden would — correctly — fail).
- **Over-investing in tick threading** — the determinism ceiling (§1) means single-sim tick parallelism has a low payoff ceiling; §2/§3/§4a are where the wins are. Don't build §4c/§4d without a profile proving the need.
